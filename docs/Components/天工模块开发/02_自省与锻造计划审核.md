# 天工自省与锻造计划审核

## 一、功能概述

天工自省是一项**定时自动化分析功能**：每晚自动分析用户近期的对话记录（短期记忆），识别其中重复出现的操作模式，判断是否值得将某个工作流封装为 CLI 工具。

与直接锻造不同，自省产出的是**锻造计划草案**，需要用户审核通过后才会进入天工的锻造队列。

```
每晚定时触发
  ↓
读取近 7 天短期记忆 + 已有工具清单
  ↓
LOW 模型分析：有没有重复工作流值得锻造？
  ↓
有 → 写入锻造计划草案到 orders/review/
无 → 记录日志，跳过
  ↓
用户审核（通过 Agent 或 Kairos 提醒）
  ↓
通过 → 移到 orders/pending/，天工下次巡查执行
拒绝 → 归档到 review/rejected/
```

---

## 二、设计动机

### 为什么需要自省？

之前天工的锻造完全是**被动的**——只有用户明确说"帮我锻造一个工具"，如意才会下达锻造令。但实际使用中，用户经常会反复执行某些操作（如转换文件格式、处理特定数据），却不会意识到"这个可以做成工具"。

自省功能让系统能够**主动发现**这些重复模式，向用户提议。

### 为什么不直接锻造？

- 锻造成本高（消耗 Coding Agent 时间和 API 费用）
- 模型的判断可能有误（误判重复、提议已有工具）
- 用户应该对自己的工具库有掌控权

所以采用**先计划、后审核**的两阶段模式。

## 三、架构设计（Agent 与 Scheduler 职责分离）

本功能的实现遵循项目的架构原则：**Agent 负责逻辑，Scheduler 负责调度**。

```
core/agent/
  ├── agent.py                  ← 主对话 Agent
  ├── kairos_agent.py           ← KAIROS 自治 Agent
  ├── memory_update_agent.py    ← 记忆更新 Agent（从 scheduler/ 移入）
  └── forge_plan_agent.py       ← [新增] 锻造计划自省 Agent

scheduler/
  ├── memory_updater.py         ← 记忆更新调度器（调用 core/agent/）
  ├── forge_plan_scheduler.py   ← [新增] 锻造计划调度器（调用 core/agent/）
  ├── cron_scheduler.py         ← 用户自定义定时任务
  ├── cron_parser.py
  └── cron_tasks.py
```

### 职责划分

| 层 | 文件 | 职责 |
|----|------|------|
| Agent 层 | `core/agent/forge_plan_agent.py` | 纯逻辑：读数据、调 LLM、返回结构化结果 |
| Scheduler 层 | `scheduler/forge_plan_scheduler.py` | 纯调度：定时触发、调用 Agent、写文件、记日志 |

这样设计的好处：
- Agent 可以被其他地方复用（比如手动触发分析）
- Scheduler 不包含任何 LLM prompt 或业务逻辑
- 新增定时 Agent 时模式清晰：`core/agent/` 写 Agent，`scheduler/` 写调度器

## 四、目录结构

在 `~/.heartclaw/tiangong/orders/` 下新增 `review/` 子目录：

```
tiangong/orders/
  pending/      ← 已审核通过，等天工执行
  processing/   ← 天工正在执行
  done/         ← 已完成
  review/       ← [新增] 自省产出的待审核锻造计划
    rejected/   ← 用户拒绝的计划归档
```

## 五、自省 Agent 详解

### 5.1 文件位置

`src/core/agent/forge_plan_agent.py`

### 5.2 核心函数

```python
async def run_forge_plan_analysis(
    llm: BaseLLMService,
    short_store: ShortMemoryStore,
    heartclaw_home: Path,
    recent_days: int = 7,
) -> ForgePlanResult
```

### 5.3 数据收集

Agent 从两个维度收集信息：

**输入数据（短期记忆）：**

- 读取近 N 天（默认 7 天）的对话记录
- 使用 `ShortMemoryStore.load_daily()` 逐日读取
- 每条记录截取前 500 字符，避免上下文过长

**去重数据（三个来源）：**

| 来源 | 路径 | 作用 |
|------|------|------|
| TianGongToolList | `skills/TianGongToolList/SKILL.md` | 最权威的已有工具清单 |
| 锻造记录 | `tiangong/forge-logs/*.md` | 已锻造/尝试过的工具 |
| 已完成锻造令 | `tiangong/orders/done/*.md` | 历史锻造令（含成功和失败） |

三源去重确保 LLM 不会提议已经存在或已经尝试过的工具。

### 5.4 LLM Prompt 设计

系统提示词要求 LLM：
1. 分析对话记录中重复出现的操作模式
2. 判断是否值得封装为 CLI 工具（重复性 + 可自动化 + 独立性）
3. 不得提议已有工具
4. 每次最多提议 1 个工具
5. 输出结构化的锻造计划草案（包含 `TOOL_NAME:` 标记行，方便解析）

### 5.5 输出格式

LLM 返回的锻造计划草案格式：

```markdown
TOOL_NAME: json-formatter

# 锻造计划：json-formatter

- 状态：待审核
- 提议时间：2026-04-15
- 提议原因：用户在 4 月 12、13、15 日多次要求格式化 JSON 文件

## 工具概述

将 JSON 文件格式化为人类可读的缩进格式，支持排序键名。

## 使用场景

用户需要美化 JSON 配置文件或 API 返回结果时使用。

## 技术方案

- 推荐实现语言：Rust
- 输入：JSON 文件路径或 stdin
- 输出：格式化后的 JSON（stdout 或覆盖写入）
- 核心逻辑：解析 JSON → serde_json::to_string_pretty → 输出
```

## 六、调度器详解

### 6.1 文件位置

`src/scheduler/forge_plan_scheduler.py`

### 6.2 配置

在 `config.json` 的 `tiangong` 节中配置定时时间：

```json
{
  "tiangong": {
    "forge_plan_schedule": "23:00"
  }
}
```

默认 23:00，比记忆更新（23:30）早半小时，避免同时运行。

### 6.3 执行流程

```
APScheduler CronTrigger 触发（每天 23:00）
  ↓
调用 forge_plan_agent.run_forge_plan_analysis()
  ↓
result.has_proposal == False → 记日志，结束
result.has_proposal == True  → 继续
  ↓
写入 orders/review/{timestamp}-{tool_name}.md
  ↓
记录日志
```

## 七、审核流程

### 7.1 Agent 侧（用户主动查看）

用户说"查看天工的锻造计划"之类的话时，Agent 执行：

1. `ListFiles` 列出 `~/.heartclaw/tiangong/orders/review/` 目录
2. `ReadFile` 读取具体的锻造计划文件
3. 展示给用户
4. 用户说"通过" → Agent 用 `Bash` 执行 `mv` 将文件从 `review/` 移到 `pending/`
5. 用户说"拒绝" → Agent 用 `Bash` 执行 `mv` 将文件移到 `review/rejected/`

### 7.2 Kairos 侧（自动提醒）

Kairos 系统提示词中增加了一条行为规则：

> 每次醒来后，检查天工锻造计划审核目录。如果有待审核的 .md 文件，在回复中提醒用户。

这样即使用户不主动查看，Kairos 也会在下次 tick 时发现并提醒。

### 7.3 审核后的流程

```
审核通过 → 文件移到 orders/pending/
  ↓
天工下次巡查（每 15 分钟）
  ↓
发现 pending/ 中有新锻造令 → 开始锻造
  ↓
（后续流程与普通锻造令一致）
```

## 八、工具变更

本功能新增了一个通用的 `WriteFile` 工具，用于文件写入操作：

| 工具名 | 参数 | 说明 |
|--------|------|------|
| WriteFile | `file_path`, `content` | 文件不存在则创建（含父目录），存在则覆盖写入 |

这个工具不仅服务于锻造计划审核，也是一个通用的文件写入能力，Agent 可以在其他场景下使用。
