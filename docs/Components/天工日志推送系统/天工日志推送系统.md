# 天工日志推送系统

## 需求背景

天工（TianGong）是一个独立的 Docker 容器（`tiangong-worker`），负责巡查锻造令并调度 Coding Agent 执行任务。前端的"天工"页面需要实时显示天工容器的运行日志，同时提供"如意调试"模式查看如意（ruyi-api）的执行日志。

### 旧方案的问题

旧方案通过 ruyi-api 进程内的 Python logging `WebSocketLogHandler` 转发内存中的日志。这些日志全部来自如意进程本身（`heartclaw.*` 命名空间），并非天工容器的日志。天工容器的日志只写入文件 `/logs/tiangong-worker.log`，前端看不到。

## 架构设计

### 整体数据流

```
┌──────────────────────────────────────────────────────────────────┐
│  Docker Volume /logs (宿主机 ~/.heartclaw/tmp/logs/services/)    │
│                                                                  │
│  tiangong-worker.log ──┐                                         │
│                        ├──→ LogFileTailer (ruyi-api 后台任务)    │
│  ruyi-api.log ─────────┘         │                               │
│                                  ▼                               │
│                        WebSocket broadcast                       │
│                     type: "container_log"                         │
│                     data: { source, timestamp, level, message }  │
│                                  │                               │
│                                  ▼                               │
│                           前端 useAppStore                        │
│                        tiangongLogs / ruyiDebugLogs              │
│                                  │                               │
│                                  ▼                               │
│                           TiangongPage                           │
│                   天工模式 / 如意调试模式                          │
└──────────────────────────────────────────────────────────────────┘
```

两个容器共享同一个日志卷（`/logs`），ruyi-api 容器可以读取天工的日志文件。

### 日志文件格式

**天工日志** (`tiangong-worker.log`)：
```
2026-04-24 10:54:26,973 [tiangong.main] INFO: Config loaded: agent=kimi, poll=30s
2026-04-24 10:54:26,974 [tiangong.engine] INFO: TianGong engine started. agent=kimi, poll_interval=30s
```

**如意日志** (`ruyi-api.log`)：
```
[2026-04-24 04:24:37] INFO heartclaw - === HeartClaw starting ===
[2026-04-24 04:24:37] INFO heartclaw.llm.factory - Creating LLM service: provider=kimi
```

后端用两套正则分别解析：

```python
# 天工格式
_RE_TIANGONG = re.compile(
    r"^(\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2},\d{3})\s+\[([^\]]+)\]\s+(\w+):\s*(.+)$"
)
# 如意格式
_RE_RUYI = re.compile(
    r"^\[(\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2})\]\s+(\w+)\s+(\S+)\s+-\s+(.+)$"
)
```

## 后端实现

### 1. LogFileTailer（实时推送）

位于 `apps/ruyi-api/src/api/routes/ws.py`。

- 启动时为每个日志文件（`tiangong-worker.log`、`ruyi-api.log`）创建一个 asyncio 后台任务
- 每个任务打开对应文件，seek 到末尾，然后每秒轮询新行（类似 `tail -f`）
- 新行经 `parse_log_line()` 解析后，通过 `ConnectionManager.broadcast()` 推送 WS 消息

```python
# WS 消息格式
{
    "type": "container_log",
    "data": {
        "source": "tiangong",              # 或 "ruyi"
        "timestamp": "2026-04-24 10:54:26.973",
        "level": "INFO",
        "message": "[tiangong.main] Config loaded: agent=kimi, poll=30s"
    }
}
```

生命周期管理：
- `main.py` → `startup()` 中调用 `get_log_file_tailer().start()` 启动
- `main.py` → `shutdown()` 中调用 `stop()` 取消所有后台任务

### 2. 历史日志 REST API（刷新加载）

位于 `apps/ruyi-api/src/api/routes/logs.py`。

```
GET /api/logs/recent?source=tiangong&lines=200
```

- 参数：`source`（`tiangong` | `ruyi`）、`lines`（1-1000，默认 200）
- 使用 `read_last_n_lines()` 从文件末尾高效读取最后 N 行
- 逐行解析为结构化 JSON 返回

用途：前端 `AppStoreProvider` mount 时调用此 API 加载历史日志，解决刷新页面后日志清空的问题。

## 前端实现

### 1. WS 类型扩展

在 `apps/web/src/hooks/useWebSocket.tsx` 中新增：

```typescript
type ContainerLog = {
  source: 'tiangong' | 'ruyi'
  timestamp: string
  level: string
  message: string
}

// WsMessage union 新增
| { type: 'container_log'; data: ContainerLog }
```

### 2. 全局状态管理

在 `apps/web/src/stores/useAppStore.tsx` 中：

- `TiangongState` 持有两个列表：`tiangongLogs`（天工日志）+ `ruyiDebugLogs`（如意日志）
- 收到 `container_log` 消息时按 `source` 分流到对应列表
- mount 时调用 `GET /api/logs/recent` 预加载历史日志

### 3. 天工页面 UI

**天工模式**（默认）：
- 工具栏显示 ALL / INFO / WARN / ERROR 级别过滤按钮
- 日志列表显示 `tiangongLogs`，支持级别筛选和搜索

**如意调试模式**：
- 点击 ERROR 按钮右侧的"如意调试"按钮切换
- 级别过滤按钮隐藏，仅保留搜索框
- 日志列表切换为 `ruyiDebugLogs`，不做级别筛选
- 按钮激活时橙色高亮，标题切换为"如意调试日志"

## 配置

| 环境变量 | 默认值 | 说明 |
|----------|--------|------|
| `HEARTCLAW_LOG_DIR` | `/logs` | 日志文件目录（Docker 卷挂载点） |

日志文件路径自动推导：
- `$HEARTCLAW_LOG_DIR/tiangong-worker.log`
- `$HEARTCLAW_LOG_DIR/ruyi-api.log`
