# 简单核心版本的定时功能

## 一、定位

从 Claude Code 的定时任务系统中提炼出**最小可用核心**。砍掉会话级内存存储、多实例锁、远程 Agent 调度、jitter 抖动等高级机制，只保留：

- **一个 JSON 文件**存储所有任务
- **一个定时循环**每秒扫描文件判断是否该触发
- **三个 Tool** 让 LLM 可以增删查任务
- **两条创建路径**：LLM 自主调用 Tool、用户 `/loop` 快捷命令

---

## 二、时间格式：标准 5 字段 Cron 表达式

### 2.1 为什么选 cron

Claude Code 选择了**标准 5 字段 cron 表达式**作为时间格式，而不是 ISO 8601 时间戳或自然语言。原因：

| 方案 | 优点 | 缺点 |
|------|------|------|
| cron 表达式 | 一个格式同时覆盖一次性和循环，生态成熟 | 用户不一定看得懂 |
| ISO 时间戳 | 一次性任务精确 | 无法表达 "每 5 分钟" |
| 自然语言 | 用户友好 | 解析模糊，LLM 不确定性高 |

Claude Code 的做法是：**底层统一用 cron，但用户永远不需要手写 cron**——LLM 负责把自然语言转成 cron，`/loop` 命令负责把简单间隔转成 cron。

### 2.2 格式定义

```
分钟  小时  日  月  星期
 M     H   DoM Mon DoW
```

五个字段，空格分隔，**本地时区**：

| 字段 | 取值范围 | 示例 |
|------|----------|------|
| 分钟 (minute) | 0-59 | `30` = 第 30 分钟 |
| 小时 (hour) | 0-23 | `14` = 下午 2 点 |
| 日 (day-of-month) | 1-31 | `1` = 每月 1 号 |
| 月 (month) | 1-12 | `6` = 六月 |
| 星期 (day-of-week) | 0-6 (0=周日) | `1-5` = 周一到周五 |

每个字段支持的语法：

| 语法 | 含义 | 示例 |
|------|------|------|
| `*` | 通配（任意值） | `* * * * *` = 每分钟 |
| `N` | 固定值 | `30 14 * * *` = 每天 14:30 |
| `*/N` | 每隔 N | `*/5 * * * *` = 每 5 分钟 |
| `N-M` | 范围 | `0 9 * * 1-5` = 工作日 9:00 |
| `N,M` | 列表 | `0,30 * * * *` = 每小时的 :00 和 :30 |

### 2.3 常见场景对应表

| 用户说的话 | cron 表达式 | 任务类型 |
|-----------|-------------|----------|
| "每 5 分钟检查一下" | `*/5 * * * *` | 循环 |
| "每小时跑一次" | `0 * * * *` | 循环 |
| "每天早上 9 点" | `0 9 * * *` | 循环 |
| "工作日下午 3 点" | `0 15 * * 1-5` | 循环 |
| "今天下午 2:30 提醒我" | `30 14 11 4 *`（4月11日） | 一次性 |
| "明天上午 10 点跑测试" | `0 10 12 4 *`（4月12日） | 一次性 |

**关键区分**：循环任务的日/月字段用 `*`（通配），一次性任务的日/月字段**固定到具体日期**。

## 三、两种任务类型

### 3.1 一次性任务（one-shot）

- `recurring: false`
- 到达触发时间后执行一次，**自动删除**
- 适合：提醒、单次延迟执行
- cron 字段里日/月/星期会固定到具体值

### 3.2 循环任务（recurring）

- `recurring: true`（默认）
- 每次到达匹配时间都会触发，直到手动删除或自动过期
- 触发后更新 `lastFiredAt`，从当前时间重新计算下次触发
- 适合：定期检查、轮询、自动化流程

## 四、存储设计

### 4.1 存储文件

只用一个 JSON 文件，路径：

```
<项目根目录>/.heartclaw/scheduled_tasks.json
```

### 4.2 文件格式

```json
{
  "tasks": [
    {
      "id": "a1b2c3d4",
      "cron": "*/5 * * * *",
      "prompt": "检查部署状态，如果有异常立即通知",
      "createdAt": 1712830000000,
      "lastFiredAt": 1712830300000,
      "recurring": true
    },
    {
      "id": "e5f6a7b8",
      "cron": "30 14 11 4 *",
      "prompt": "提醒用户下午 3 点有产品评审会议",
      "createdAt": 1712820000000,
      "recurring": false
    }
  ]
}
```

### 4.3 CronTask 数据结构

简化版只保留核心字段：

```typescript
type CronTask = {
  id: string           // 8 位随机 hex（UUID 前 8 位）
  cron: string         // 5 字段 cron 表达式（本地时间）
  prompt: string       // 触发时要执行的 prompt
  createdAt: number    // 创建时间（epoch ms）
  lastFiredAt?: number // 上次触发时间（仅循环任务有值）
  recurring: boolean   // true=循环 false=一次性
}
```

与 Claude Code 原版的区别——**砍掉了**：
- `permanent`：系统级永不过期标记（简单版不需要）
- `durable`：运行时标记区分内存/磁盘（简单版只有磁盘）
- `agentId`：队友路由（简单版没有多 Agent）

### 4.4 CRUD 操作

```typescript
// 读取全部任务（文件不存在/格式错误 → 空数组）
async function readCronTasks(): Promise<CronTask[]>

// 覆盖写入全部任务
async function writeCronTasks(tasks: CronTask[]): Promise<void>

// 添加一个任务 → 返回 ID
async function addCronTask(cron: string, prompt: string, recurring: boolean): Promise<string>

// 按 ID 删除任务
async function removeCronTasks(ids: string[]): Promise<void>

// 更新 lastFiredAt（循环任务触发后调用）
async function markCronTasksFired(ids: string[], firedAt: number): Promise<void>
```

**读取的健壮性**（照搬 Claude Code 的做法）：
- 文件不存在 → 返回 `[]`
- JSON 解析失败 → 返回 `[]`
- 单条任务字段缺失 → 跳过该条，不影响其他
- cron 表达式无效 → 跳过该条

## 五、三个核心 Tool

Claude Code 中定时任务相关的工具（Tool）一共有 **3 个**：

| Tool 名称 | 作用 | 只读 | 输入参数 |
|-----------|------|------|----------|
| **CronCreate** | 创建定时任务 | 否 | `cron`, `prompt`, `recurring` |
| **CronList** | 列出所有任务 | 是 | 无 |
| **CronDelete** | 删除指定任务 | 否 | `id` |

### 5.1 CronCreate — 创建任务

这是最核心的 Tool，LLM 通过它创建任务。

**输入参数：**

```typescript
{
  cron: string      // 标准 5 字段 cron 表达式
  prompt: string    // 到期时要执行的 prompt
  recurring: boolean // true=循环（默认） false=一次性
}
```

**验证逻辑（依次检查）：**

1. cron 表达式语法是否合法
2. 在未来一年内是否有匹配的时间点
3. 任务总数是否超过上限（50 个）

**执行逻辑：**

1. 生成 8 位 hex ID（`randomUUID().slice(0, 8)`）
2. 构造任务对象，追加到 JSON 文件
3. 启动/唤醒调度器
4. 返回 `{ id, humanSchedule, recurring }`

**返回给 LLM 的消息示例：**

```
循环：Scheduled recurring job a1b2c3d4 (Every 5 minutes). Use CronDelete to cancel.
一次性：Scheduled one-shot task e5f6a7b8 (Apr 11 at 2:30 PM). It will fire once then auto-delete.
```

**给 LLM 的 Prompt 引导**（关键设计，简化版建议保留）：

Claude Code 在 CronCreate 的 Tool Prompt 中写了一段重要的引导：

> "Avoid the :00 and :30 minute marks when the task allows it. Every user who asks for '9am' gets `0 9`, and every user who asks for 'hourly' gets `0 *`. When the user's request is approximate, pick a minute that is NOT 0 or 30."

这是为了避免所有用户的任务都在整点触发导致 API 拥堵。简单版虽然没有 jitter 机制，但在 Prompt 里引导 LLM 错开时间是零成本的优化。

### 5.2 CronList — 列出任务

**输入参数：** 无

**执行逻辑：** 读取 JSON 文件，返回所有任务

**返回格式：**

```
a1b2c3d4 — Every 5 minutes (recurring): 检查部署状态，如果有异常立即通知
e5f6a7b8 — Apr 11 at 2:30 PM (one-shot): 提醒用户下午 3 点有会议
```

如果没有任务：
```
No scheduled jobs.
```

### 5.3 CronDelete — 删除任务

**输入参数：**

```typescript
{
  id: string  // CronCreate 返回的 Job ID
}
```

**验证逻辑：** 检查该 ID 是否存在

**执行逻辑：** 从 JSON 文件中移除该条任务

**返回消息：**
```
Cancelled job a1b2c3d4.
```

## 六、两种创建方式

### 6.1 方式一：用户输入 → Agent 自主调用 CronCreate

**流程：**

```
用户："每 5 分钟检查一下部署状态"
  ↓
LLM 理解意图 → 决定调用 CronCreate
  ↓
LLM 构造参数：
  {
    cron: "*/5 * * * *",
    prompt: "检查部署状态，如果有异常立即通知",
    recurring: true
  }
  ↓
CronCreate 执行 → 写入 JSON → 返回结果
  ↓
LLM 回复："已设置每 5 分钟检查部署状态（任务 ID: a1b2c3d4）"
```

**特点：**
- **用户不需要知道 cron 格式**——完全用自然语言
- LLM 负责把自然语言转成 cron 表达式
- LLM 负责判断是一次性还是循环
- LLM 负责构造 prompt 内容（可以比用户原话更精确）

**用户说的话 vs LLM 构造的参数：**

| 用户说 | LLM 构造的 cron | recurring |
|--------|----------------|-----------|
| "每 5 分钟检查部署" | `*/5 * * * *` | true |
| "每天早上 9 点给我总结" | `57 8 * * *` | true |
| "半小时后提醒我开会" | `00 18 11 4 *` | false |
| "明天 10 点跑一下测试" | `0 10 12 4 *` | false |

注意 LLM 会把 "每天早上 9 点" 转成 `57 8`（8:57）而不是 `0 9`——这是 Tool Prompt 中的引导，避开整点。

### 6.2 方式二：/loop 快捷命令

**格式：**

```
/loop [间隔] <prompt>
```

**流程：**

```
用户输入：/loop 5m 检查部署状态
  ↓
loop 技能解析（Skill 层，不是 Tool 层）
  ↓
提取出：interval = "5m", prompt = "检查部署状态"
  ↓
构造 Prompt 指令给 LLM：
  "把 5m 转成 cron 表达式，调用 CronCreate..."
  ↓
LLM 调用 CronCreate：
  { cron: "*/5 * * * *", prompt: "检查部署状态", recurring: true }
  ↓
CronCreate 执行 → 返回结果
  ↓
LLM **立即执行一次** prompt（不等第一次 cron 触发）
```

**支持的间隔格式：**

| 格式 | 含义 | 转换为 cron |
|------|------|-------------|
| `5m` | 每 5 分钟 | `*/5 * * * *` |
| `30m` | 每 30 分钟 | `*/30 * * * *` |
| `1h` | 每 1 小时 | `0 * * * *` |
| `2h` | 每 2 小时 | `0 */2 * * *` |
| `1d` | 每 1 天 | `0 0 * * *` |
| `30s` | 30 秒 | `*/1 * * * *`（向上取整到 1 分钟）|

**输入解析规则（优先级从高到低）：**

1. **前置间隔**：第一个 token 匹配 `^\d+[smhd]$` → 那就是间隔，剩余是 prompt
   - `/loop 5m check deploy` → interval=5m, prompt="check deploy"

2. **尾部 every 子句**：结尾匹配 `every <N><unit>` → 提取间隔，剩余是 prompt
   - `/loop check deploy every 20m` → interval=20m, prompt="check deploy"

3. **默认**：没有间隔 → 默认 10 分钟，整体是 prompt
   - `/loop check deploy` → interval=10m, prompt="check deploy"

**示例：**

```
/loop 5m /babysit-prs           → 每 5 分钟执行 /babysit-prs
/loop 30m check the deploy      → 每 30 分钟执行 "check the deploy"
/loop 1h /standup 1             → 每小时执行 /standup 1
/loop check the deploy          → 默认 10 分钟执行 "check the deploy"
/loop check deploy every 20m    → 每 20 分钟执行 "check deploy"
```

### 6.3 两种方式的对比

| 维度 | Agent 自主调用 | /loop 命令 |
|------|---------------|-----------|
| **入口** | 用户自然语言对话 | 固定的斜杠命令 |
| **谁解析时间** | LLM 自主判断 | /loop 按规则解析间隔 → 再让 LLM 转 cron |
| **任务类型** | 一次性或循环（LLM 判断） | **只创建循环任务** |
| **用户需要知道格式吗** | 不需要，自然语言即可 | 需要知道 `Nm/Nh/Nd` 间隔格式 |
| **prompt 来源** | LLM 根据对话上下文构造 | 用户直接写 |
| **创建后行为** | 等待下次触发 | **立即执行一次** + 等待后续触发 |
| **适用场景** | "提醒我..."、"每天 9 点..."等自然语言 | "每 5 分钟跑这个命令" 这种明确循环 |

**关键区别：`/loop` 创建后会立即执行一次 prompt**，而 Agent 自主创建的任务要等到第一次 cron 触发才执行。这是因为 `/loop` 的语义是 "从现在开始循环做这件事"，用户期望立即开始。

### 6.4 底层统一

无论哪种方式，**最终都归结到同一个调用**：

```
CronCreate Tool
  输入: { cron, prompt, recurring }
  输出: { id, humanSchedule, recurring }
```

`/loop` 不是独立的 Tool，它是一个 **Skill**（技能），本质上是构造了一段 Prompt 指令丢给 LLM，让 LLM 去调用 CronCreate。流程差异只在 "到达 CronCreate 之前"，到达之后完全一样。

## 七、调度器执行

### 7.1 最简调度器

简化版的调度器只需要一个 `setInterval`：

```typescript
const CHECK_INTERVAL_MS = 1000  // 每秒检查一次

function createSimpleScheduler(onFire: (task: CronTask) => void) {
  let tasks: CronTask[] = []
  let nextFireAt = new Map<string, number>()
  let timer: ReturnType<typeof setInterval> | null = null

  async function loadTasks() {
    tasks = await readCronTasks()
    // 为新任务计算下次触发时间
    for (const t of tasks) {
      if (!nextFireAt.has(t.id)) {
        const anchor = t.lastFiredAt ?? t.createdAt
        const next = nextCronRunMs(t.cron, anchor)
        nextFireAt.set(t.id, next ?? Infinity)
      }
    }
    // 清理已删除任务的缓存
    for (const id of nextFireAt.keys()) {
      if (!tasks.find(t => t.id === id)) nextFireAt.delete(id)
    }
  }

  function check() {
    const now = Date.now()

    for (const task of tasks) {
      const next = nextFireAt.get(task.id)
      if (next === undefined || now < next) continue

      // 触发！
      onFire(task)

      if (task.recurring) {
        // 循环：从当前时间重新计算下次触发
        const newNext = nextCronRunMs(task.cron, now) ?? Infinity
        nextFireAt.set(task.id, newNext)
        // 更新 lastFiredAt 写回文件
        void markCronTasksFired([task.id], now)
      } else {
        // 一次性：删除任务
        nextFireAt.delete(task.id)
        void removeCronTasks([task.id])
      }
    }
  }

  return {
    async start() {
      await loadTasks()
      timer = setInterval(check, CHECK_INTERVAL_MS)
      // 定期重新加载文件（简单版用轮询代替 chokidar 文件监听）
      setInterval(loadTasks, 5000) // 每 5 秒重新读取文件
    },
    stop() {
      if (timer) clearInterval(timer)
    }
  }
}
```

### 7.2 触发后做什么

任务触发时，调用 `onFire(task)` 回调。在 REPL 中的实现：

```typescript
onFire: (task) => {
  // 1. 显示一条系统消息告诉用户
  showMessage(`Running scheduled task (${formatTime(new Date())})`)
  // 2. 把 task.prompt 加入命令队列，优先级 "later"（不抢占用户输入）
  enqueueCommand({ value: task.prompt, priority: 'later' })
}
```

### 7.3 调度核心逻辑

**nextCronRunMs 的工作方式：**

```
给定 cron="*/5 * * * *"，from=14:32:15
  → 从 14:33 开始逐分钟扫描
  → 14:33 不匹配（33 % 5 ≠ 0）
  → 14:34 不匹配
  → 14:35 匹配！
  → 返回 14:35:00 的 epoch ms
```

**为什么循环任务从 now 而不是 next 重新调度：**

如果 5 分钟循环任务在 14:35 该触发，但系统在 14:40 才检查到（因为之前在忙），如果从 14:35 计算下次是 14:40（又立即触发），会导致快速追赶。从 `now`（14:40）计算则下次是 14:45，间隔保持 5 分钟。

## 九、Cron 表达式的人类可读转换

这是提升用户体验的关键。Claude Code 的 `cronToHuman()` 函数覆盖了常见模式：

| cron | 转换结果 |
|------|---------|
| `*/5 * * * *` | Every 5 minutes |
| `0 * * * *` | Every hour |
| `30 * * * *` | Every hour at :30 |
| `0 */2 * * *` | Every 2 hours |
| `0 9 * * *` | Every day at 9:00 AM |
| `0 9 * * 1` | Every Monday at 9:00 AM |
| `0 9 * * 1-5` | Weekdays at 9:00 AM |

简化版只需覆盖以上这些模式，不匹配的直接返回原始 cron 字符串。

## 十、完整调用链路示意

### 路径 A：自然语言 → Agent 创建

```
用户："每天早上帮我看看有没有新的 PR"
  │
  ├─ LLM 理解意图
  │   → 这是循环任务，每天早上
  │   → 构造 prompt："检查是否有新的 Pull Request，如果有则列出标题和链接"
  │   → 选择 cron: "57 8 * * *"（8:57，避开整点）
  │
  ├─ LLM 调用 CronCreate
  │   输入: { cron: "57 8 * * *", prompt: "检查是否有...", recurring: true }
  │
  ├─ CronCreate 执行
  │   ├─ 验证 cron 合法性 ✓
  │   ├─ 验证未来有匹配时间 ✓
  │   ├─ 验证任务数未超限 ✓
  │   ├─ 生成 ID: "a1b2c3d4"
  │   ├─ 写入 scheduled_tasks.json
  │   └─ 返回 { id: "a1b2c3d4", humanSchedule: "Every day at 8:57 AM", recurring: true }
  │
  └─ LLM 回复用户
      "已设置，每天早上 8:57 会自动检查新 PR（任务 ID: a1b2c3d4）"
```

### 路径 B：/loop 命令创建

```
用户输入：/loop 5m /babysit-prs
  │
  ├─ loop Skill 解析
  │   → 匹配规则 1（前置间隔）
  │   → interval: "5m", prompt: "/babysit-prs"
  │
  ├─ 构造 Prompt 指令给 LLM
  │   "把 5m 转成 cron，调用 CronCreate，然后立即执行 prompt"
  │
  ├─ LLM 调用 CronCreate
  │   输入: { cron: "*/5 * * * *", prompt: "/babysit-prs", recurring: true }
  │
  ├─ CronCreate 执行
  │   ├─ 验证 → 写入 → 返回
  │
  ├─ LLM 确认
  │   "已设置每 5 分钟执行 /babysit-prs（任务 ID: a1b2c3d4）"
  │
  └─ LLM 立即执行 /babysit-prs（不等第一次 cron）
```

### 路径 C：定时触发

```
调度器 check()（每秒执行）
  │
  ├─ 当前时间 now = 14:35:00
  ├─ 任务 a1b2c3d4 的 nextFireAt = 14:35:00
  ├─ now >= nextFireAt → 触发！
  │
  ├─ onFire(task)
  │   ├─ 显示系统消息 "Running scheduled task (Apr 11 2:35pm)"
  │   └─ enqueue(task.prompt)  // 加入队列等 REPL 空闲时执行
  │
  ├─ recurring = true
  │   ├─ 计算新的 nextFireAt: 14:40:00
  │   └─ 写回 lastFiredAt: 14:35:00
  │
  └─ 队列排到该 prompt 时
      → LLM 执行 "检查部署状态..."
      → 输出结果给用户
```
