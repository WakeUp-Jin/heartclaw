# Claude Code 定时任务系统（Cron Scheduling System）

## 一、系统概述

Claude Code 内置了一套完整的定时任务系统（内部代号 **Kairos**），允许用户通过 cron 表达式调度 prompt 的执行。该系统支持**一次性任务**和**循环任务**两种模式，支持**持久化存储**（文件）和**会话级存储**（内存）两种生命周期，并且通过分布式锁机制解决了多实例并发的问题。

### 核心特性

- 标准 5 字段 cron 表达式解析（minute hour day-of-month month day-of-week）
- 本地时区执行（用户说 "9am" 就是本地 9am）
- 两种任务类型：一次性（one-shot）和循环（recurring）
- 两种存储模式：磁盘持久化（durable）和会话级内存（session-only）
- 多实例互斥的调度锁（scheduler lock）
- 基于 GrowthBook 的远程配置抖动（jitter）参数
- 错过任务检测与恢复机制
- 循环任务自动过期（默认 7 天）
- 雷群效应（thundering herd）缓解

### 数据流

```
用户输入 /loop 5m check deploy
        ↓
    loop.ts（技能解析）
        ↓
    CronCreateTool.call()
        ↓
    addCronTask()  →  写入 .claude/scheduled_tasks.json（durable）
                   →  写入 STATE.sessionCronTasks（session-only）
        ↓

        ↓
    cronScheduler.enable()

        ↓
    到达触发时间 → onFire(prompt) → 加入命令队列
```

## 三、定时任务的创建

### 3.1 创建入口

定时任务有以下几个创建入口：

#### （1）CronCreateTool — 主要 Tool 接口

文件：`tools/ScheduleCronTool/CronCreateTool.ts`

这是模型（LLM）直接调用的工具接口，接收以下参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| `cron` | string | 标准 5 字段 cron 表达式（本地时间） |
| `prompt` | string | 到期时要执行的 prompt |
| `recurring` | boolean | 是否循环（默认 true） |
| `durable` | boolean | 是否持久化到磁盘（默认 false） |

**创建流程：**

1. **输入校验**（`validateInput`）：
   - 使用 `parseCronExpression()` 验证 cron 语法
   - 调用 `nextCronRunMs()` 确认在未来一年内存在匹配时间
   - 检查任务总数不超过 `MAX_JOBS = 50`
   - 队友（teammate）创建的任务不允许设置 `durable: true`

2. **执行创建**（`call`）：
   - 通过 `isDurableCronEnabled()` 判断持久化功能开关
   - 调用 `addCronTask()` 写入存储
   - 调用 `setScheduledTasksEnabled(true)` 启动调度器

#### （2）/loop 斜杠命令 — 便捷循环调度

文件：`skills/bundled/loop.ts`

提供了用户友好的命令行接口，格式为 `/loop [interval] <prompt>`：

- `interval` 支持：`Ns`（秒，向上取整到分钟）、`Nm`（分钟）、`Nh`（小时）、`Nd`（天）
- 自动将间隔转换为 cron 表达式
- 默认间隔 10 分钟
- 创建后立即执行一次

示例：
```
/loop 5m /babysit-prs          → */5 * * * *
/loop check the deploy         → 默认 10m → */10 * * * *
/loop check the deploy every 20m → */20 * * * *
```

#### （3）/schedule 远程 Agent 调度

文件：`skills/bundled/scheduleRemoteAgents.ts`

这是一个独立的远程调度系统，与本地 cron 不同：
- 通过 `RemoteTriggerTool` 调用 Anthropic 云端 API
- 每次触发在云端创建完全隔离的远程会话（CCR）
- cron 表达式使用 **UTC 时间**（不是本地时间）
- 最小间隔 1 小时

### 3.2 Task ID 生成

```typescript
const id = randomUUID().slice(0, 8) // 8 位 hex 字符
```

取 UUID 前 8 个十六进制字符，在 50 个任务的上限内足够唯一。这个短 ID 也会被用于 jitter 的确定性哈希计算。

### 3.3 功能开关

文件：`tools/ScheduleCronTool/prompt.ts`

```
isKairosCronEnabled()  →  总开关（编译时 AGENT_TRIGGERS + 运行时 tengu_kairos_cron）
isDurableCronEnabled() →  持久化子开关（运行时 tengu_kairos_cron_durable）
```

- `AGENT_TRIGGERS`：编译时 feature flag，用于死代码消除
- `tengu_kairos_cron`：GrowthBook 运行时开关，5 分钟刷新，默认 `true`（GA 状态）
- `CLAUDE_CODE_DISABLE_CRON`：本地环境变量覆盖
- `tengu_kairos_cron_durable`：持久化磁盘写入的子开关

## 四、定时任务的存储

### 4.1 存储方式

定时任务有两种存储方式：

#### （1）文件存储（Durable）

- **路径**：`<project_root>/.claude/scheduled_tasks.json`
- **格式**：

```json
{
  "tasks": [
    {
      "id": "a1b2c3d4",
      "cron": "*/5 * * * *",
      "prompt": "check the deploy status",
      "createdAt": 1712345678000,
      "lastFiredAt": 1712346000000,
      "recurring": true,
      "permanent": false
    }
  ]
}
```

- 通过 `writeCronTasks()` 写入，使用 `mkdir` 确保 `.claude/` 目录存在
- 空任务列表仍然写入空文件（而非删除），确保 chokidar 文件监听器能感知变化
- 写入时自动剥离 `durable` 和 `agentId` 等运行时属性

#### （2）内存存储（Session-only）

- 存储位置：`bootstrap/state.ts` 中的 `STATE.sessionCronTasks` 数组
- 进程退出后自动消失
- 不需要文件监听，调度器每次 tick 直接从内存读取
- 默认模式——大多数 "提醒我 5 分钟后..." 的场景

### 4.2 CronTask 类型定义

```typescript
type CronTask = {
  id: string           // 8 位 hex ID
  cron: string         // 5 字段 cron 表达式（本地时间）
  prompt: string       // 触发时要执行的 prompt
  createdAt: number    // 创建时间（epoch ms）
  lastFiredAt?: number // 上次触发时间（仅循环任务）
  recurring?: boolean  // 是否循环
  permanent?: boolean  // 是否免除自动过期（仅系统任务）
  durable?: boolean    // 运行时标记，false = 会话级
  agentId?: string     // 运行时标记，创建者队友 ID
}
```

### 4.3 读取操作

```typescript
// 读取文件中的任务
async function readCronTasks(dir?: string): Promise<CronTask[]>

// 同步检查是否有任务（用于启动判断）
function hasCronTasksSync(dir?: string): boolean

// 合并文件任务 + 会话任务
async function listAllCronTasks(dir?: string): Promise<CronTask[]>
```

`readCronTasks()` 的健壮性设计：
- 文件不存在或无法访问 → 返回空数组
- JSON 解析失败 → 返回空数组
- 单条任务格式错误 → 跳过该条，不影响其他任务
- cron 表达式无效 → 跳过该条并记录调试日志

### 4.4 写入操作

```typescript
// 覆盖写入（创建、删除、更新 lastFiredAt）
async function writeCronTasks(tasks: CronTask[], dir?: string): Promise<void>

// 追加任务
async function addCronTask(cron, prompt, recurring, durable, agentId?): Promise<string>

// 删除任务（文件 + 内存双路径）
async function removeCronTasks(ids: string[], dir?: string): Promise<void>

// 批量更新 lastFiredAt
async function markCronTasksFired(ids: string[], firedAt: number, dir?: string): Promise<void>
```

## 五、定时任务的读取

### 5.1 调度器的任务加载

调度器通过两条路径获取任务：

1. **文件任务**：调度器启动时通过 `readCronTasks()` 加载，并通过 chokidar 监听 `scheduled_tasks.json` 文件变化自动重新加载
2. **会话任务**：每次 `check()` tick 通过 `getSessionCronTasks()` 从内存直接读取

### 5.2 chokidar 文件监听

```typescript
watcher = chokidar.watch(path, {
  persistent: false,        // 不阻止进程退出
  ignoreInitial: true,      // 忽略初始扫描
  awaitWriteFinish: {       // 等待写入稳定
    stabilityThreshold: 300 // 300ms 无变化视为写入完成
  },
  ignorePermissionErrors: true,
})
watcher.on('add', () => void load(false))
watcher.on('change', () => void load(false))
watcher.on('unlink', () => { tasks = []; nextFireAt.clear() })
```

- `add`/`change` 事件触发 `load(false)`（非初始加载，不处理错过任务）
- `unlink` 事件清空所有任务和调度计划
- `persistent: false` 确保文件监听器不阻止进程退出

### 5.3 CronList Tool

文件：`tools/ScheduleCronTool/CronListTool.ts`

用户（通过 LLM）可以调用 `CronList` 工具查看所有定时任务：

- 调用 `listAllCronTasks()` 合并文件和会话任务
- 队友只能看到自己创建的任务
- 返回每个任务的 id、cron、人类可读描述、prompt、是否循环、是否持久化

## 六、Cron 表达式解析

### 6.1 解析器

文件：`utils/cron.ts`

支持标准 5 字段 cron 子集：

```
minute(0-59) hour(0-23) day-of-month(1-31) month(1-12) day-of-week(0-6, 0=Sunday)
```

**支持的语法：**
- 通配符：`*`
- 固定值：`N`
- 步进：`*/N`
- 范围：`N-M`、`N-M/S`
- 列表：`N,M,...`
- day-of-week 中 `7` 作为 Sunday 的别名

**不支持的语法：** L、W、?、名称别名（MON/JAN 等）

### 6.2 下次运行时间计算

```typescript
function computeNextCronRun(fields: CronFields, from: Date): Date | null
```

- 从 `from` 的下一分钟开始逐分钟向前搜索
- 最多搜索 366 天 × 24 小时 × 60 分钟
- 遵循标准 cron 语义：当 day-of-month 和 day-of-week 同时约束时，满足任一即匹配（OR 语义）
- DST 处理：春季跳过的小时自然跳过（该小时不存在于本地时间中），秋季回退只触发一次

### 6.3 人类可读转换

```typescript
function cronToHuman(cron: string, opts?: { utc?: boolean }): string
```

支持的模式：
- `*/N * * * *` → "Every N minutes"
- `M * * * *` → "Every hour at :MM"
- `M */N * * *` → "Every N hours at :MM"
- `M H * * *` → "Every day at H:MMam/pm"
- `M H * * D` → "Every DayName at H:MMam/pm"
- `M H * * 1-5` → "Weekdays at H:MMam/pm"

## 七、定时任务的执行

### 7.1 调度器核心

文件：`utils/cronScheduler.ts`

调度器通过 `createCronScheduler()` 工厂函数创建，核心结构：

```typescript
type CronScheduler = {
  start: () => void       // 启动调度
  stop: () => void        // 停止调度
  getNextFireTime: () => number | null  // 下次触发时间
}
```

### 7.2 启动流程

```
start()
  ├── 检查 dir 参数（daemon 路径直接 enable）
  ├── 检查 hasCronTasksSync()（有任务则自动启用）
  ├── getScheduledTasksEnabled() ?
  │     ├── true  → enable()
  │     └── false → 启动 1s 轮询等待标志翻转
  │
  └── enable()
        ├── import chokidar（动态导入）
        ├── tryAcquireSchedulerLock()（获取调度锁）
        │     ├── 成功 → isOwner = true
        │     └── 失败 → 启动 5s 锁探测定时器
        ├── load(true)（初始加载 + 错过任务检测）
        ├── chokidar.watch() 监听文件变化
        └── setInterval(check, 1000ms)（启动主循环）
```

### 7.3 check() 主循环（每秒执行一次）

```
check()
  ├── isKilled?() → 被远程杀死则立即返回
  ├── isLoading() → REPL 正在处理查询则跳过
  │
  ├── 遍历文件任务（仅当 isOwner 时）
  │   └── process(task, isSession=false)
  │
  ├── 遍历会话任务（始终处理）
  │   └── process(task, isSession=true)
  │
  ├── 批量写回 lastFiredAt
  └── 清理已消失任务的调度缓存
```

### 7.4 process() 单任务处理

```
process(task, isSession)
  ├── filter 过滤 → 不匹配则跳过
  ├── inFlight 检查 → 正在处理中则跳过
  │
  ├── 首次见到?
  │   ├── 循环任务 → jitteredNextCronRunMs(cron, lastFiredAt ?? createdAt, id)
  │   └── 一次性任务 → oneShotJitteredNextCronRunMs(cron, createdAt, id)
  │
  ├── now < nextFireAt? → 未到时间，跳过
  │
  ├── 触发！
  │   ├── onFireTask(task) 或 onFire(prompt)
  │   ├── 记录分析事件 tengu_scheduled_task_fire
  │   │
  │   ├── 检查是否过期（isRecurringTaskAged）
  │   │
  │   ├── 循环且未过期 → 从 now 重新调度（不从 next，避免快速追赶）
  │   ├── 会话级一次性/过期 → removeSessionCronTasks()（同步内存移除）
  │   └── 文件级一次性/过期 → removeCronTasks()（异步文件删除 + inFlight 防重）
```

### 7.5 REPL 中的执行

文件：`hooks/useScheduledTasks.ts`

REPL 通过 React Hook 封装调度器：

```typescript
useScheduledTasks({ isLoading, assistantMode, setMessages })
```

触发时的处理：
1. 创建 `SystemScheduledTaskFireMessage` 系统消息显示在 UI
2. 通过 `enqueuePendingNotification()` 将 prompt 加入命令队列
3. 命令优先级为 `later`（不会抢占用户输入）
4. 标记 `isMeta: true`（对 UI 隐藏）
5. 标记 `workload: WORKLOAD_CRON`（API 可以按低 QoS 处理）

对于队友（teammate）创建的任务：
- 通过 `task.agentId` 路由到对应队友的消息队列
- 如果队友已不存在，自动清理孤儿任务

### 7.6 SDK/-p 模式的执行

文件：`cli/print.ts`

无头模式下直接使用 `createCronScheduler`：

```typescript
cronScheduler = createCronScheduler({
  onFire: prompt => enqueue({ mode: 'prompt', value: prompt, ... }),

```

## 八、抖动（Jitter）机制

### 8.1 问题背景

当大量用户都设置了 `0 9 * * *`（每天 9 点）或 `0 * * * *`（每小时整点）时，所有任务同时触发会造成 API 的瞬时压力（雷群效应）。

### 8.2 抖动策略

#### 循环任务抖动 — 正向延迟

```typescript
function jitteredNextCronRunMs(cron, fromMs, taskId, cfg): number | null
```

- 在标准触发时间上**加上**一个延迟
- 延迟 = `jitterFrac(taskId) × recurringFrac × (下次间隔时长)`
- 上限：`recurringCapMs`（默认 15 分钟）
- 效果：每小时任务在 :00 到 :06 之间分散

#### 一次性任务抖动 — 反向提前

```typescript
function oneShotJitteredNextCronRunMs(cron, fromMs, taskId, cfg): number | null
```

- 只在触发时间的分钟数是 `oneShotMinuteMod` 的倍数时生效（默认 30 → 只处理 :00 和 :30）
- 在标准触发时间**减去**一个提前量
- 提前量 = `oneShotFloorMs + jitterFrac(taskId) × (oneShotMaxMs - oneShotFloorMs)`
- 效果：用户说 "3pm 提醒我" 会在 2:58:30 ~ 3:00:00 之间触发，用户无感知

#### 确定性哈希

```typescript
function jitterFrac(taskId: string): number {
  return parseInt(taskId.slice(0, 8), 16) / 0x1_0000_0000
}
```

将 8 位 hex ID 转换为 [0, 1) 的浮点数，跨重启保持稳定。

### 8.3 远程可调参数

文件：`utils/cronJitterConfig.ts`

通过 GrowthBook `tengu_kairos_cron_config` 配置，60 秒刷新：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `recurringFrac` | 0.1 | 循环任务抖动比例 |
| `recurringCapMs` | 15min | 循环任务抖动上限 |
| `oneShotMaxMs` | 90s | 一次性任务最大提前量 |
| `oneShotFloorMs` | 0 | 一次性任务最小提前量 |
| `oneShotMinuteMod` | 30 | 只对 :00/:30 生效 |
| `recurringMaxAgeMs` | 7 天 | 循环任务自动过期时间 |

运维可通过推送配置在事故期间扩大抖动窗口，无需重启客户端。

## 九、多实例调度锁

### 9.1 问题背景

当多个 Claude 会话在同一项目目录运行时，如果都执行调度器，同一个一次性任务会被触发多次。

### 9.2 锁机制

文件：`utils/cronTasksLock.ts`

- **锁文件路径**：`<project_root>/.claude/scheduled_tasks.lock`
- **锁内容**：`{ sessionId, pid, acquiredAt }`
- **创建方式**：`writeFile(path, body, { flag: 'wx' })` — 原子性 O_EXCL 创建

### 9.3 获取锁流程

```
tryAcquireSchedulerLock()
  ├── 尝试 O_EXCL 创建 → 成功 = 获取锁
  │
  ├── 文件已存在 → 读取锁内容
  │   ├── sessionId 匹配（幂等） → 更新 PID → 返回 true
  │   ├── PID 仍在运行 → 返回 false
  │   └── PID 已死亡 → 删除旧锁 → 再次尝试 O_EXCL
  │       ├── 成功 → 获取锁
  │       └── 失败 → 另一个会话赢得竞争
  │
  └── ENOENT → 创建 .claude/ 目录后重试一次
```

### 9.4 锁探测

未获取锁的会话每 5 秒探测一次：

```typescript
lockProbeTimer = setInterval(() => {
  void tryAcquireSchedulerLock(lockOpts).then(owned => {
    if (owned) {
      isOwner = true
      clearInterval(lockProbeTimer)
    }
  })
}, 5000) // LOCK_PROBE_INTERVAL_MS
```

### 9.5 锁释放

- 正常退出：通过 `registerCleanup()` 注册清理回调
- 异常退出：PID 不再存在，其他会话通过探测恢复锁

### 9.6 会话任务不受锁影响

会话级任务（`durable: false`）存在于进程内存中，天然不会被其他会话看到，因此无需锁保护。调度器对这些任务始终处理，不检查 `isOwner`。

## 十、错过任务检测

### 10.1 检测逻辑

```typescript
function findMissedTasks(tasks: CronTask[], nowMs: number): CronTask[] {
  return tasks.filter(t => {
    const next = nextCronRunMs(t.cron, t.createdAt)
    return next !== null && next < nowMs
  })
}
```

从 `createdAt` 计算第一个触发时间，如果该时间已过去，则认为任务被错过。

### 10.2 处理方式

- 仅在调度器**初始加载**时检测（`load(true)`）
- 仅处理**一次性任务**（循环任务由 `check()` 正常调度，会在首次 tick 立即触发）
- 生成通知消息，要求模型先用 `AskUserQuestion` 工具询问用户是否执行
- 错过的任务立即从文件中删除

### 10.3 通知格式

```
The following one-shot scheduled task was missed while Claude was not running.
It has already been removed from .claude/scheduled_tasks.json.

Do NOT execute this prompt yet. First use the AskUserQuestion tool to ask
whether to run it now. Only execute if the user confirms.

[Every day at 2:30 PM, created 4/5/2026, 10:00:00 AM]
```

prompt 内容用代码围栏包裹，围栏长度比 prompt 中最长的反引号序列多 1 个，防止 prompt 注入。

## 十一、自动过期

### 11.1 过期规则

```typescript
function isRecurringTaskAged(t: CronTask, nowMs: number, maxAgeMs: number): boolean {
  if (maxAgeMs === 0) return false
  return Boolean(t.recurring && !t.permanent && nowMs - t.createdAt >= maxAgeMs)
}
```

- 仅影响循环任务
- `permanent: true` 的任务免除过期（系统级任务，如 assistant mode 的 catch-up/morning-checkin/dream）
- 默认 7 天后过期
- 过期任务会最后触发一次，然后删除
- 过期时间通过 `recurringMaxAgeMs` 远程可调

### 11.2 设计动机

Cron 是多日会话（multi-day sessions）的主要驱动力（p99 运行时间从 61 分钟增长到 53 小时），无限循环会让堆泄漏无限累积。7 天的默认限制覆盖了 "每小时检查我的 PR 持续一周" 这样的工作流，同时限制了最坏情况下的会话生命周期。

## 十二、关键设计决策总结

| 决策 | 理由 |
|------|------|
| 本地时间而非 UTC | 用户说 "9am" 就是本地 9am，无需心智转换 |
| 默认 session-only | 大多数场景是 "5 分钟后提醒我"，无需持久化 |
| 8 位 hex ID | 50 个任务上限内足够唯一，同时用于确定性 jitter |
| O_EXCL 原子锁 | 无需外部依赖即可实现多进程互斥 |
| 1 秒检查间隔 | cron 最小粒度是 1 分钟，1 秒足够精确 |
| chokidar 文件监听 | 外部编辑 JSON 也能被感知 |
| jitter 正向/反向分离 | 循环任务可以延迟无影响，一次性任务不能延迟但可提前 |
| 批量 markCronTasksFired | 同一 tick 多任务触发只做一次读-修改-写 |
| permanent 标记 | assistant mode 任务不能重建（writeIfMissing 跳过已存在文件） |
| isKilled 轮询 | GrowthBook 杀开关可以停止已运行的调度器，不只是阻止新的 |
