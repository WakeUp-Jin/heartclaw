# 心跳机制（Heartbeat）与定时任务（Cron）的区别

## 一、一句话定位

- **Cron 定时任务** = **闹钟**：到点就执行，不管有没有必要
- **Heartbeat 心跳** = **值班巡检员**：定时醒来看看有没有事要做，没事就继续睡

---

## 二、核心区别详解

### 2.1 触发逻辑：确定性 vs 智能判断

**Cron — 确定性触发**

```
时间到了 → 立即执行 → 完毕
```

Cron 是"无脑"的。你设定了"每天 9 点执行"，到了 9 点它就执行，不会问"今天有没有必要执行"。任务内容在创建时就固定了，每次执行的都是同一个 `payload.message`。

**Heartbeat — 两阶段智能触发**

```
时间到了 → Phase 1: LLM 阅读 HEARTBEAT.md 判断要不要做 → skip/run
                                                            │
                                    skip → 什么都不做，继续睡  │
                                    run  → Phase 2: 执行任务 ←┘
                                            │
                                            └→ Phase 3: LLM 评估结果是否值得通知用户
```

Heartbeat 在执行前**多了一层 LLM 决策**。它先把待办清单（HEARTBEAT.md）交给 LLM 看，让 LLM 判断当前是否有需要行动的任务。只有 LLM 认为"有活干"时，才真正进入执行阶段。

**这意味着**：如果 HEARTBEAT.md 里写着"关注 BTC 价格，跌到 5 万以下通知我"，心跳机制每 30 分钟检查一次，但只有价格真的跌到 5 万以下时 LLM 才会决定 `run`，其他时候都 `skip`。

### 2.2 任务来源：创建时固定 vs 随时可变

**Cron — 任务内容在创建时锁定**

```python
# 创建时确定了 message，后续每次执行都是这个内容
cron.add_job(
    name="weather-check",
    schedule=CronSchedule(kind="every", every_ms=600000),
    message="检查天气并报告",
)
```

要修改任务内容，必须通过 API（`update_job`）或删除后重建。

**Heartbeat — 任务来自文件，随时可编辑**

```markdown
## Active Tasks

- 监控 BTC 价格，跌破 50000 通知我
- 每次检查一下 GitHub 仓库有没有新 issue
- 如果明天天气预报有雨，提前提醒我带伞
```

直接用文本编辑器（或让 Agent 用 `edit_file` 工具）修改 `HEARTBEAT.md` 文件即可。不需要任何 API 调用，下次心跳醒来时自然会读到新内容。

**这意味着**：Heartbeat 的任务描述可以是模糊的、条件性的、开放式的，因为 LLM 每次都会重新理解文件内容。

### 2.3 任务管理方式

| 维度 | Cron | Heartbeat |
|------|------|-----------|
| **创建** | `CronTool(action="add", ...)` 或代码 API | 编辑 HEARTBEAT.md 文件 |
| **查看** | `CronTool(action="list")` | 读取 HEARTBEAT.md |
| **删除** | `CronTool(action="remove", job_id=...)` | 从 HEARTBEAT.md 删除对应行 |
| **修改** | `CronService.update_job(...)` | 编辑 HEARTBEAT.md 对应行 |
| **管理工具** | 结构化的 API + Tool | 纯文本文件编辑 |

### 2.4 调度精度

| 维度 | Cron | Heartbeat |
|------|------|-----------|
| **最小精度** | 毫秒级（`at_ms`）/ 秒级（`every_ms`） | 固定间隔（默认 30 分钟） |
| **时间表达** | cron 表达式 / 固定间隔 / 精确时刻 | 仅固定间隔 |
| **时区支持** | 支持 IANA 时区 | 支持（传入 timezone 参数） |
| **调度灵活度** | 高（"工作日 9 点"、"每 5 分钟"、"明天下午 3 点"） | 低（只能是固定间隔） |

### 2.5 Session 管理

**Cron — 每个任务独立 session**

```python
resp = await agent.process_direct(
    reminder_note,
    session_key=f"cron:{job.id}",  # 每个 job 有独立的会话
)
```

每个定时任务维护自己的对话历史，任务之间互不干扰。

**Heartbeat — 共享单一 session + 定期裁剪**

```python
resp = await agent.process_direct(
    tasks,
    session_key="heartbeat",  # 所有心跳共享一个会话
)
# 裁剪历史，防止无限增长
session.retain_recent_legal_suffix(keep_recent_messages)
```

所有心跳执行共享一个 `heartbeat` session，且会定期裁剪旧消息，保持会话有界。这意味着心跳有一定的"短期记忆"——它能参考最近几次的检查结果，但不会记住很久以前的事。

### 2.6 通知策略

| 维度 | Cron | Heartbeat |
|------|------|-----------|
| **默认行为** | 根据 `deliver` 字段决定 | 仅当有值得报告的结果时 |
| **执行后评估** | 有（`evaluate_response`） | 有（`evaluate_response`） |
| **额外的执行前过滤** | 无 | 有（Phase 1 LLM 决策） |
| **无外部渠道时** | 任务仍然执行 | 不发送通知（`if channel == "cli": return`） |

## 三、适用场景对比

### 3.1 适合用 Cron 的场景

| 场景 | 为什么用 Cron |
|------|-------------|
| "每天早上 9 点提醒我站会" | 时间精确，内容固定，到点就通知 |
| "每 10 分钟检查 GitHub star 数量并报告" | 固定间隔，每次都要执行同样的操作 |
| "明天下午 3 点提醒我开会" | 一次性精确时间点 |
| "工作日下午 5 点发送日报" | 需要 cron 表达式的复杂调度 |
| "每 2 小时执行系统健康检查" | 固定周期的自动化任务 |

**共同特征**：时间明确、任务内容固定、到点就要执行。

### 3.2 适合用 Heartbeat 的场景

| 场景 | 为什么用 Heartbeat |
|------|-------------------|
| "关注 BTC 价格，跌到 5 万以下通知我" | 条件性任务，不是每次都需要行动 |
| "有新的 PR 就帮我 review" | 只有新 PR 出现时才需要执行 |
| "如果明天天气有雨，提前提醒我" | 需要 LLM 判断当前条件是否满足 |
| "帮我盯着竞品网站，有重大更新就通知" | 模糊的监控任务，需要 LLM 理解 |
| 同时管理多个模糊的后台任务 | 一个文件管理多个任务，灵活增删 |

**共同特征**：任务描述模糊或有条件、需要 LLM 理解和判断、可能经常变化。

### 3.3 AGENTS.md 中的官方指引

Nanobot 在 `AGENTS.md` 模板中明确指出：

> "When the user asks for a recurring/periodic task, update `HEARTBEAT.md` instead of creating a one-time cron reminder."

**设计意图**：
- 精确定时提醒 → 用 Cron
- 模糊的持续性后台任务 → 用 Heartbeat

## 四、执行流程对比图

### Cron 执行流程

```
[用户/Agent 创建任务]
        │
        ▼
[CronService 存储任务到 jobs.json]
        │
        ▼
[Timer 到期] ────────────────────────► [直接执行]
                                           │
                                           ▼
                                  [agent.process_direct()]
                                           │
                                           ▼
                                  [evaluate_response()]
                                           │
                                    值得通知？
                                   /         \
                                  是          否
                                  │           │
                          [推送到用户渠道]   [静默]
```

### Heartbeat 执行流程

```
[用户/Agent 编辑 HEARTBEAT.md]
        │
        ▼
[定时器到期（默认 30 分钟）]
        │
        ▼
[读取 HEARTBEAT.md] ─────► [Phase 1: LLM 判断]
                                   │
                            skip / run ?
                           /            \
                         skip           run
                          │              │
                      [结束，继续睡]       ▼
                                   [Phase 2: agent.process_direct()]
                                         │
                                         ▼
                                   [evaluate_response()]
                                         │
                                   值得通知？
                                  /         \
                                 是          否
                                 │           │
                          [推送到用户渠道]   [静默]
```

**关键差异在入口**：Cron 到期就执行，Heartbeat 到期后先让 LLM 判断要不要执行。

## 五、资源消耗对比

| 维度 | Cron | Heartbeat |
|------|------|-----------|
| **每次触发的 LLM 调用** | 1 次（执行任务） + 可选 1 次（评估通知） | 1 次（Phase 1 决策）+ 可选 1 次（执行）+ 可选 1 次（评估） |
| **skip 时的开销** | 不存在 skip，到期必执行 | 仅 1 次轻量 LLM 调用（Phase 1） |
| **存储开销** | jobs.json（结构化，含运行历史） | HEARTBEAT.md（纯文本） |
| **内存开销** | 每个 job 一个 session | 共享一个 session（定期裁剪） |

Heartbeat 的设计在"大多数时候没事做"的场景下更高效——Phase 1 是一次轻量的 LLM 调用（短 prompt + 工具调用），如果判断 skip，就不需要启动完整的 Agent Loop。

## 六、协作关系

Cron 和 Heartbeat 并不是互斥的，它们在 Nanobot 中**并行运行**，各司其职：

```
gateway 启动
  ├── cron.start()       # Cron 调度器开始工作
  ├── heartbeat.start()  # 心跳服务开始工作
  ├── agent.run()        # Agent 消息循环
  └── channels.start_all()

运行时：
  ├── Cron Timer ──────► 精确时间触发任务
  ├── Heartbeat Timer ──► 周期性智能巡检
  └── 两者共享：
       ├── 同一个 AgentLoop（agent.process_direct）
       ├── 同一个 LLM Provider
       ├── 同一个 MessageBus（消息投递）
       └── 同一个 evaluate_response（通知评估）
```

**它们甚至可以互相配合**：
- 用户可以让 Heartbeat 监控某个条件，当条件满足时，Agent 在执行过程中通过 CronTool 创建一个精确的 Cron 提醒
- Dream（记忆巩固）作为 Cron 的系统任务运行，但 Heartbeat 检查时也能感知到 Agent 的记忆状态变化

## 七、总结

| | Cron 定时任务 | Heartbeat 心跳 |
|---|---|---|
| **本质** | 时间驱动的任务执行器 | LLM 驱动的智能巡检器 |
| **触发条件** | 时间到了 | 时间到了 + LLM 判断有事做 |
| **任务来源** | API 创建，内容固定 | 文件编辑，内容动态 |
| **执行前过滤** | 无 | 有（LLM Phase 1 决策） |
| **调度精度** | 高（毫秒级，支持 cron 表达式） | 低（固定间隔，默认 30 分钟） |
| **管理方式** | 结构化 API / Tool | 编辑 Markdown 文件 |
| **Session** | 每任务独立 | 共享 + 定期裁剪 |
| **最佳场景** | 精确定时、固定内容 | 模糊监控、条件触发 |
| **类比** | 闹钟 | 值班巡检员 |

**选择建议**：
- 如果你能明确回答"什么时间做什么事" → 用 **Cron**
- 如果你的需求更像"帮我盯着，有情况就告诉我" → 用 **Heartbeat**
