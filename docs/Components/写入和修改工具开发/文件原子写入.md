# 文件原子写入

## 1. 概述

文件原子写入是一种确保**写入操作要么完整成功，要么完全不发生**的机制。核心思路是：不直接覆盖目标文件，而是先把内容写入一个临时文件，然后通过一次系统级的原子 `rename` 操作将临时文件替换为目标文件。

### 为什么要做原子写入

如果直接覆盖原文件（`writeFileSync(target, content)`），流程是：

1. 打开目标文件 → 截断为 0 字节
2. 逐块写入新内容
3. 关闭文件

如果在第 2 步时进程崩溃、断电或磁盘故障，目标文件将处于**只写了一半的中间状态**，内容永久损坏且不可恢复。

原子写入通过 `rename` 把"覆盖文件"压缩成一个**不可中断的系统调用**，从根本上消除中间状态。

## 2. 实现机制

`utils/file.ts:386-438`

### 2.1 生成唯一临时文件路径

```ts
const tempPath = `${targetPath}.tmp.${process.pid}.${Date.now()}`
```

包含进程 ID 和时间戳，确保并发场景下不会冲突。

### 2.2 写入临时文件并刷盘

```ts
fsWriteFileSync(tempPath, content, { encoding, flush: true })
```

`flush: true` 确保数据从操作系统缓存真正落到磁盘（fsync），避免 rename 后数据还在内存里没写出去。

### 2.3 保留原文件权限

```ts
if (targetExists && targetMode !== undefined) {
  chmodSync(tempPath, targetMode)
}
```

临时文件的默认权限通常是 `0o644`（所有人可读），如果原文件是 `0o600`（仅所有者可读写），直接 rename 会导致权限被覆盖。因此 rename 之前先把临时文件的权限改成与原文件一致。

### 2.4 原子 rename 替换

```ts
fs.renameSync(tempPath, targetPath)
```

这是最关键的一步。在 POSIX 系统上，`rename()` 是一个**原子操作**。

---

## 3. 为什么 rename 是原子的

操作系统层面，`rename` 不是"复制内容 + 删除旧文件"，而是**只改目录里的文件名到 inode 的映射**。

```
rename 前：
  /project/main.ts                → inode A（旧内容）
  /project/main.ts.tmp.123.456    → inode B（新内容）

renameSync(tempPath, targetPath) 后：
  /project/main.ts                → inode B（新内容）
  /project/main.ts.tmp.123.456    → 路径消失，无残留
  inode A（旧内容）               → 引用计数归零，文件系统回收
```

整个操作只有一条指令：把目录中 `main.ts` 指向的 inode 从 A 换成 B。这个操作在操作系统内核中是不可分割的，**不存在"换到一半"的中间状态**。

> **Windows 差异**：Windows 上的 `rename` 在目标存在时会先删除目标再重命名，不是严格原子。但 Node.js 的 `renameSync` 在 Windows 上会调用 `MoveFileEx` 并尽可能保证原子性。跨文件系统的 rename 可能不支持原子替换。

## 4. 符号链接处理

```ts
let targetPath = filePath
try {
  const linkTarget = fs.readlinkSync(filePath)
  targetPath = isAbsolute(linkTarget)
    ? linkTarget
    : resolve(dirname(filePath), linkTarget)
} catch {
  // ENOENT 或 EINVAL（不是 symlink）——保持原路径
}
```

如果目标文件是符号链接，写入操作会穿透 symlink，写到 symlink 指向的**真实目标文件**，同时保留 symlink 本身不被破坏。

## 5. 异常处理与 Fallback

`utils/file.ts:439-477`

### 5.1 原子写入失败时的处理

```ts
try {
  fs.renameSync(tempPath, targetPath)
} catch (atomicError) {
  // 1. 记录监控事件
  logEvent('tengu_atomic_write_error', {})

  // 2. 清理临时文件
  try {
    fs.unlinkSync(tempPath)
  } catch (cleanupError) {
    logForDebugging(`Failed to clean up temp file: ${cleanupError}`)
  }

  // 3. 回退到非原子写入
  fsWriteFileSync(targetPath, content, { encoding, flush: true })
}
```

| 步骤 | 说明 |
|------|------|
| 记录事件 | `tengu_atomic_write_error` 用于监控原子写入失败率 |
| 清理临时文件 | `unlinkSync(tempPath)` 删除残留临时文件 |
| 非原子回退 | 直接覆盖目标文件，至少保证写入成功（牺牲原子性） |

### 5.2 临时文件清理的时机

| 场景 | 临时文件状态 |
|------|--------------|
| rename 成功 | 临时文件路径已变成目标文件，**无需清理** |
| rename 失败 | catch 块中调用 `unlinkSync` **显式清理** |
| 写入临时文件失败 | 异常直接向上抛出，**不会走到 rename**，临时文件可能残留（但进程退出后通常无影响） |

## 7. 与 TOCTOU 防护的关系

| 机制 | 防护对象 | 层面 |
|------|----------|------|
| **TOCTOU 防护** | "基于过期快照的写入"导致的数据覆盖 | 逻辑层面（读 → 写 之间的时间窗口） |
| **原子写入** | "写入过程中崩溃"导致的文件损坏 | 物理层面（写操作的完整性） |

两者互补：

- **TOCTOU** 确保模型写的方向是对的（文件没被外部修改过）
- **原子写入** 确保写的过程不会把文件搞坏（即使进程崩溃也不会产生残缺的中间文件）

在 `FileEditTool` 和 `FileWriteTool` 中，**先经过 TOCTOU 校验，再通过原子写入落盘**，形成完整的写入安全链路。
