# 新的安全权限设计（Bash）

## 1. 目标与原则

本文档定义 Bash 工具的新安全权限主链路：

1. 解析
2. 静态检查
3. 权限验证
4. 模型验证
5. 容器验证
6. 执行

核心原则：

- 先静态、后权限、再模型增益，最后执行。
- 静态检查与显式规则是硬约束，模型只能增益，不能覆盖硬拒绝。
- 解析失败或无法静态证明安全时，默认 fail-closed（ask/deny）。

## 2. 六大节点设计

### 2.1 节点一：解析（Parse）

职责：把原始命令转换为可分析结构（AST/IR），并判定可分析性。

- 入口：`parseCommandRaw` + `parseForSecurityFromAst`
- 关键输出：`simple | too-complex | parse-unavailable`
- 前置建议：对原始命令字符串做 pre-parse gate（控制字符/Unicode 空白/反斜杠空白）后再进入 AST 解析。命中即 `too-complex`，可直接跳过 `parseCommandRaw`。

源码位置：

- `utils/bash/ast.ts:400` `parseForSecurityFromAst`
- `tools/BashTool/bashPermissions.ts:1663` `bashToolHasPermission` 中解析分流
- `utils/bash/ast.ts:186` `DANGEROUS_TYPES`（触发 too-complex 的结构类型）

设计要点：

- `simple` 才进入深度静态检查。
- `cd+git` 风险应在解析后基于执行关系判定，不建议在解析前仅用字符串共现（同时出现 `cd` 与 `git`）直接定为危险。
- `too-complex` 必须先尝试 deny 早退，再 ask。
- `parse-unavailable` 进入 Legacy 语法预检，不可直接放行。

---

### 2.2 节点二：静态检查（Static Security Checks）

职责：在不依赖用户交互的前提下，识别结构风险、语义风险、路径风险、组合风险。

这是规则最多、最核心的一层。

#### 2.2.1 静态检查总览

静态检查由三块组成：

1. AST 语义静态检查（命令语义与执行原语）
2. 结构/操作符静态检查（subshell、管道、组合）
3. 路径与重定向静态检查（越界与敏感路径）

---

#### 2.2.2 静态检查明细（逐条对应源码 + 原因）

以下每条都来自项目现有实现，可直接映射。

1. 控制字符与 Unicode 空白拒绝（too-complex）
- 位置：`utils/bash/ast.ts:400`（`parseForSecurityFromAst` 内）
- 原因：解析器与 shell 实际分词可能出现差异，容易产生隐蔽注入与绕过。
- 执行时机建议：当前实现位于 `parseForSecurityFromAst`。从防御和性能角度，建议上移为“解析前检查”（在 `parseCommandRaw` 前执行），命中后直接返回 `too-complex`。

2. 危险结构类型（DANGEROUS_TYPES）触发 too-complex
- 位置：`utils/bash/ast.ts:186`（`DANGEROUS_TYPES`）
- 原因：如 process substitution、控制流等结构难以在当前规则下完整静态证明安全。

3. wrapper 去壳后一致性检查（timeout/nice/env/stdbuf/nohup/time）
- 位置：`utils/bash/ast.ts:2213`（`checkSemantics` 开头 wrapper strip）
- 原因：若只检查外层 wrapper，会漏掉真实执行命令（典型绕过）。

4. 命令名健壮性检查（空名/占位符/片段）
- 位置：`utils/bash/ast.ts:2388`-`2419`
- 原因：空命令名、占位符命令名、片段化命令常导致“解析视图”和“执行视图”不一致。

5. eval-like builtin 拦截
- 位置：`utils/bash/ast.ts:2086`（`EVAL_LIKE_BUILTINS`），`utils/bash/ast.ts:2626`
- 原因：这类 builtin 会再次解释参数，等价动态执行入口。

6. zsh 危险 builtin 拦截
- 位置：`utils/bash/ast.ts:2060`（`ZSH_DANGEROUS_BUILTINS`），`utils/bash/ast.ts:2619`
- 原因：`zmodload` 等可启用绕过面（模块能力扩展），不应在自动放行链路执行。

7. 数组下标执行面（flag 触发）
- 位置：`utils/bash/ast.ts:2143`（`SUBSCRIPT_EVAL_FLAGS`），`utils/bash/ast.ts:2428`
- 原因：某些 builtin 在 NAME 位置会算术求值，可能触发 `$(...)` 执行。

8. read/unset 裸位置 NAME 下标执行面
- 位置：`utils/bash/ast.ts:2182`（`BARE_SUBSCRIPT_NAME_BUILTINS`），`utils/bash/ast.ts:2504`
- 原因：不需要显式危险 flag 也会把位置参数当 NAME 解释。

9. `[[ ... ]]` 算术比较两侧操作数检查
- 位置：`utils/bash/ast.ts:2477`
- 原因：两侧操作数可能触发算术求值链，属于隐式执行入口。

10. Shell 关键字误解析防御
- 位置：`utils/bash/ast.ts:2191`（`SHELL_KEYWORDS` 引入），`utils/bash/ast.ts:2547`
- 原因：关键字作为 argv[0] 往往代表 AST 误解析，必须 fail-closed。

11. newline + `#` 注释错位防御（参数隐藏）
- 位置：`utils/bash/ast.ts:2204`（`NEWLINE_HASH_RE`），`utils/bash/ast.ts:2561`-`2584`
- 原因：可制造“上游看见参数、下游忽略参数”的错位，影响路径校验正确性。

12. `jq system()` 与危险 flag 拦截
- 位置：`utils/bash/ast.ts:2588`-`2616`
- 原因：`jq` 可成为执行与任意文件读取桥接点。

13. `/proc/*/environ` 敏感访问拦截
- 位置：`utils/bash/ast.ts:2197`（`PROC_ENVIRON_RE`），`utils/bash/ast.ts:2658`-`2674`
- 原因：可能泄露进程环境中的密钥与凭据。

14. 复杂结构操作符检查（subshell/command group）
- 位置：`tools/BashTool/bashCommandHelpers.ts:181`，`tools/BashTool/bashCommandHelpers.ts:218`-`223`
- 原因：结构化复合命令可隐藏执行边界，必须先拦。

15. 管道分段递归检查（按段递归）
- 位置：`tools/BashTool/bashCommandHelpers.ts:23`，`tools/BashTool/bashCommandHelpers.ts:49`，`tools/BashTool/bashCommandHelpers.ts:243`
- 原因：`|` 两侧需要分别进入完整权限链（规则、路径、语义检查）后再聚合结果，避免“只看整串”漏检某一段的风险。

16. 解析后 `cd + git` 执行关系判定（建议）
- 位置：`tools/BashTool/bashPermissions.ts:2144`，`tools/BashTool/bashPermissions.ts:2202`，`utils/bash/ast.ts:65`
- 原因：`cd` 与 `git` 是否构成风险依赖控制操作符语义。应在解析后按执行关系判定（如 `;`/`&&`/顺序 list 共享 cwd 风险更高），不建议在解析前仅靠字符串共现直接拦截。对 `too-complex`/`parse-unavailable` 保持 fail-closed（ask）。

17. process substitution 路径层拦截（Legacy 路径）
- 位置：`tools/BashTool/pathValidation.ts:1013`（`checkPathConstraints`），`1021`-`1038`
- 原因：`>(cmd)`/`<(cmd)` 可在重定向之外隐式执行写入。

18. 重定向目标安全检查（含危险 expansion）
- 位置：`tools/BashTool/pathValidation.ts:1046`-`1061`，`924`-`1003`
- 原因：重定向目标若可展开/越界，可能写入任意文件。

19. 危险删除路径拦截（rm/rmdir）
- 位置：`tools/BashTool/pathValidation.ts:66`-`108`，`728`-`736`
- 原因：`rm -rf /` 类灾难操作必须强制人工确认。

20. `cd + write` 组合路径不确定性拦截
- 位置：`tools/BashTool/pathValidation.ts:630`-`655`，`930`-`945`
- 原因：复合命令里 cwd 变化后路径解析不确定，自动判定不可靠。

21. `--` 终止符与 flag 解析健壮性（防参数错判）
- 位置：`tools/BashTool/pathValidation.ts:110`-`139` 及多处 extractors 注释
- 原因：若把 `--` 后参数误当 flag 丢弃，会漏检真实危险路径。

22. 路径命令 wrapper 去壳后再校验
- 位置：`tools/BashTool/pathValidation.ts:840`-`846`，`888`-`894`
- 原因：防止 `timeout 10 rm ...` 这类外壳绕过路径检查。

23. Legacy 注入安全网（仅 AST 不可用）
- 位置：`tools/BashTool/bashPermissions.ts:2078`-`2140`；`tools/BashTool/bashSecurity.ts:2426`
- 原因：在 AST 不可用时，仍需对已知注入/误解析模式做兜底。

24. 安全 heredoc 例外重检
- 位置：`tools/BashTool/bashSecurity.ts:521`（`stripSafeHeredocSubstitutions`）
- 原因：减少误报同时不放松对真正注入模式的拦截。

25. 子命令 fanout 上限防护
- 位置：`tools/BashTool/bashPermissions.ts:103`，`2164`-`2178`
- 原因：防止超大拆分触发 CPU 饥饿/DoS 风险。

### 2.3 节点三：权限验证（Rule/Mode Permission）

职责：执行确定性权限策略，不依赖模型。

- 工具级规则：exact/prefix/wildcard deny/ask/allow
- 模式策略：bypass/dontAsk/plan/auto 等

源码位置：

- `utils/permissions/permissions.ts:1158` `hasPermissionsToUseToolInner`
- `utils/permissions/permissions.ts:1169`-`1206` 工具级 deny/ask
- `utils/permissions/permissions.ts:1262`-`1319` mode 与 ask 汇总
- `tools/BashTool/bashPermissions.ts:991` `bashToolCheckExactMatchPermission`

设计要点：

- deny 优先级最高。
- ask 规则必须是硬约束，不被模型覆盖。
- tool.checkPermissions 只返回工具域判定，不直接处理 UI。

---

### 2.4 节点四：模型验证（Model-Assisted Validation）

职责：在 `ask/passthrough` 上做增益判断，降低交互频率。

源码位置：

- `tools/BashTool/bashPermissions.ts:1459` `buildPendingClassifierCheck`
- `tools/BashTool/bashPermissions.ts:1497` `startSpeculativeClassifierCheck`
- `tools/BashTool/bashPermissions.ts:1605` `executeAsyncClassifierCheck`

设计约束：

- 只能输出 `allow` 或 `no_change`。
- 不得覆盖静态硬阻断、显式 deny、显式 ask。
- 高置信 allow 才可自动批准。

---

### 2.5 节点五：容器验证（Execution Isolation Planning）

职责：在“允许执行”后决定执行隔离形态。

源码位置：

- `tools/BashTool/shouldUseSandbox.ts:130` `shouldUseSandbox`
- `tools/BashTool/shouldUseSandbox.ts:21` `containsExcludedCommand`
- `tools/BashTool/BashTool.tsx:896` 执行参数 `shouldUseSandbox`

设计要点：

- 这是执行规划，不是权限判定替代。
- 即使容器可用，前置静态/权限仍必须先通过。

---

### 2.6 节点六：执行（Run & Audit）

职责：按最终决策执行命令并留下审计信息。

源码位置：

- `tools/BashTool/BashTool.tsx:881` 执行 `exec(...)`
- `services/tools/toolExecution.ts:916` 执行前统一权限检查
- `hooks/toolPermission/permissionLogging.ts:181` 权限决策审计

设计要点：

- 执行层不改写安全决策。
- 必须记录决策来源（rule/model/user/hook）和关键 reason code。

---

## 3. 最终裁决优先级（建议固定）

建议在实现层显式区分“解析前预检”与“解析”：
`raw pre-check -> parse -> static checks -> permission -> model -> container -> execute`

建议统一优先级：

1. 静态硬拒绝（deny/ask）
2. 显式权限规则（deny > ask > allow）
3. 模型增益（仅 ask/passthrough -> allow）
4. 容器规划（决定 sandbox/unsandbox）
5. 执行

---

## 4. 错误返回规范（给模型与上层调度）

建议每个拒绝都返回结构化字段：

- `stage`: parse/static/permission/model/container
- `reason_code`: 机器可读
- `message`: 人类可读（避免可绕过细节）
- `action_hint`: 建议动作（如请求批准、缩小路径范围）

示例：

```json
{
  "stage": "static",
  "reason_code": "DANGEROUS_REMOVAL_PATH",
  "message": "Dangerous rm operation on critical path requires manual approval",
  "action_hint": "Use a narrower target path inside allowed workspace"
}
```

---

## 5. 实施建议（落地顺序）

1. 先固化六层边界与返回结构。
2. 将静态检查规则注册表化（规则ID、风险等级、源码链接）。
3. 保持模型层“只增益不降级”的硬约束。
4. 容器层仅做执行规划，不再参与权限裁决逻辑。
5. 建立回归测试矩阵：每条静态规则至少一条命中用例和一条非命中用例。
