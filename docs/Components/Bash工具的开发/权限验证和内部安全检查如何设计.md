# 权限验证和内部安全检查如何设计

## 1. 结论先行

你提出的两种运行模式是对的，但要补两条关键约束：

- 不管运行在容器还是宿主机，`静态安全检查`都必须是硬前置。
- 不管运行在容器还是宿主机，主权限判定都应统一走 `checkPermissions`，而不是散落在函数执行内部。

建议统一主链路：

1. 解析
2. 静态安全检查
3. 权限验证（规则/模式）
4. 模型验证（可选增益）
5. 执行隔离判断（容器/宿主机）
6. 执行与审计

---

## 2. 对你两个判断的分析

### 2.1 方案 A：代码运行在 Docker 容器中

你的观点：默认可高权限、减少权限验证，只拦危险命令。

可行，但不能简化为“无需权限验证”或“完全弱化”：

- 仍需要静态检查：防命令注入、防路径越界、防危险组合命令。
- 仍需要权限规则：容器可能挂载宿主目录、注入密钥、开放网络。
- 可以减少人工确认频率：更多走自动 allow，但必须保留硬阻断规则。

建议：

- 容器模式允许“弱交互、强静态、统一 checkPermissions”。
- 即：用户弹窗可以少，但静态硬规则不能少，权限入口不能省。

### 2.2 方案 B：代码运行在宿主机上

你的观点：需要安全验证，用户作为最后一层可 allow。

基本正确，但要区分“软规则”和“硬规则”：

- 软规则：可由用户确认后执行（ask -> allow）。
- 硬规则：不允许用户覆盖（deny，不可强行 allow）。

否则会出现“系统判定明确高危，用户误点后直接破坏宿主机”的风险。

## 3. 推荐的双模式设计

## 3.1 统一接口（必须）

所有工具在调度层统一调用：

- `tool.checkPermissions(input, context)`

不要把主权限链路散落在工具 `call()` 内部。

职责分层：

- 调度层：统一编排（规则、模式、模型、交互、审计）。
- 工具层：提供工具域静态分析和领域规则判断（通过 `checkPermissions` 暴露）。
- 执行层：只做运行时保障与实际执行。

---

## 3.2 模式一：Container-First（容器优先）

适用：CI、隔离执行、低信任代码处理。

策略：

- 静态硬规则：启用且不可覆盖。
- 权限规则：默认宽松，可自动放行常规命令。
- 用户交互：最小化（减少 ask）。
- 执行：默认容器内执行。

仍要硬阻断的典型项：

- 明确注入原语（eval-like/不可静态分析结构）。
- 危险删除路径。
- 敏感信息读取路径。
- 结构过于复杂且不可证明安全。

---

## 3.3 模式二：Host-Guarded（宿主机防护）

适用：开发机、生产跳板机、真实账号环境。

策略：

- 静态硬规则：启用且不可覆盖。
- 权限规则：严格（deny/ask 明确分层）。
- 用户交互：高风险动作必须显式确认。
- 执行：允许宿主机执行，但优先沙箱或最小权限执行器。

---

## 4. “硬规则 vs 软规则”分级（关键）

## 4.1 硬规则（Hard Deny / Hard Ask）

定义：不允许被模型覆盖；其中 Hard Deny 不允许被用户覆盖。

建议硬规则来源：

- 解析不可证安全（too-complex 的高风险子类）。
- 明确执行原语（eval/source 等）。
- 危险删除/敏感目录写入。
- 涉及凭据/环境泄露路径。

处理：

- Hard Deny：直接拒绝执行。
- Hard Ask：必须人工确认，且模型不能自动批准。

## 4.2 软规则（Soft Ask）

定义：默认 ask，可被用户允许；模型可在高置信时自动放行。

示例：

- 常规读操作但越出当前工作目录。
- 非破坏性命令组合，风险可解释。

---

## 5. 模型验证层应该怎么放

模型层定位：只做“增益”，不做“主判官”。

允许：

- `ask/passthrough -> allow`（高置信）

禁止：

- 覆盖显式 `deny`
- 覆盖硬规则产生的 `deny/hard ask`

推荐输出：

- `allow`
- `no_change`

不要用 `false`，避免和 `deny` 语义混淆。

---

## 6. checkPermissions 与函数内部的边界（最终版）

你提到“宿主机场景统一写入到工具的 checkPermissions，然后放入调度节点执行”，这个方向是正确的，而且容器模式也应如此。

推荐实现：

- 调度层统一调用 `hasPermissionsToUseTool -> tool.checkPermissions`。
- `checkPermissions` 内根据 `executionProfile`（container/host）走不同强度策略。
- 工具 `call()` 只关心运行时保障与执行，不再承载主权限编排逻辑。

工具函数内部建议保留的少量校验/保障：

- 超时与取消。
- 资源限制（CPU/内存/输出大小/并发）。
- 沙箱落地与执行器参数拼装。
- 进程树清理、异常兜底、结果归档。

即：

- 一个统一权限入口（checkPermissions）
- 两套策略参数化（container/host）
- 一套执行时保障（函数内部）
- 一套审计与错误返回规范

---

## 7. 错误返回规范（给模型/调度/UI）

每次拒绝建议返回结构化对象：

- `stage`: parse/static/permission/model/container
- `severity`: hard|soft
- `reason_code`: 机器可读
- `message`: 人类可读
- `can_user_override`: true|false
- `suggestion`: 可选修复建议

示例：

```json
{
  "stage": "static",
  "severity": "hard",
  "reason_code": "DANGEROUS_REMOVAL_PATH",
  "message": "Dangerous rm target requires blocking",
  "can_user_override": false,
  "suggestion": "Use a narrower path inside workspace"
}
```

## 8. 最终建议（针对你的方案）

你的设计方向正确，建议定稿为：

- 容器模式：降低交互，不降低静态与硬规则。
- 宿主机模式：保留用户最终确认，但必须有不可覆盖硬规则。
- 权限链路统一放调度层执行，工具通过 `checkPermissions` 提供领域判断。
- 函数内部保持干净，仅保留运行时保障，不做主权限决策。

一句话版本：

- 不是“容器=不用权限，宿主机=全靠用户”。
- 而是“容器与宿主机共享硬安全基线，差异只在策略强度与交互方式”。

---

## 9. 你这轮问题的直接答案（可引用）

1. Docker 运行时，权限验证不能取消，只能弱化交互。
2. 宿主机运行时，强交互模式下把主安全验证放在 `checkPermissions` 是最佳方案。
3. 最终最优解：无论 Docker 还是宿主机，都走 `checkPermissions` 做主判定；函数内部只放超时、资源限制、沙箱执行、进程清理等运行时保障。
