# 权限验证与内部安全检查的设计

> 本文档阐明 Bash 工具（及类似高权限工具）中**安全检查（Security Check）**与**权限验证（Permission Check）**的边界、优先级与适用场景。

---

## 1. 核心区分

| 维度 | 内部安全检查（Security Check） | 权限验证（Permission Check） |
|------|------------------------------|----------------------------|
| **性质** | 硬约束（Hard Constraint） | 软约束 / 用户策略（Policy） |
| **可覆盖性** | **不可覆盖** | **用户可覆盖**（allow/deny/ask 规则） |
| **目标** | 防止已知灾难性后果 | 控制用户工作流中的命令范围 |
| **典型例子** | `rm -rf /`、`eval`、`zmodload`、路径越界 | `Bash(git commit:*)`、`Bash(npm run:*)` |
| **失败行为** | `deny`（直接拒绝执行） | `ask`（弹窗询问，用户可改规则） |
| **依赖环境** | 任何环境都必须有 | 视隔离强度而定 |

**一句话总结**：
> 安全检查是**底线**，权限规则是**偏好**。安全检查永远优先，且不可被用户规则或模型覆盖。

---

## 2. 两种运行环境的策略差异

### 2.1 环境 A：Docker / 强隔离容器

**假设**：代码运行在受控容器中，文件系统、网络、进程空间均已隔离。即使执行了危险命令，破坏范围也被限制在容器内。

**设计策略**：
- **权限验证**：可大幅简化或完全跳过
  - 无需 exact/prefix/wildcard 规则库
  - 无需 ask 弹窗
  - 无需分类器增益
- **内部安全检查**：**必须保留**
  - 依然要拦截 `rm -rf /`（防止容器内数据意外全清）
  - 依然要拦截 `eval` / `exec`（防止元数据泄露、横向移动）
  - 依然要做路径越界检查（防止写到挂载的宿主机卷）

**执行流程**：

```
输入命令
  │
  ▼
[安全检查]
  ├── 不通过 → 直接 deny（硬拒绝）
  └── 通过 ──► 直接执行（无需权限弹窗）
```

> 注意：如果容器挂载了宿主机敏感目录（如 `~/.ssh`），路径约束检查必须把这些挂载点也纳入保护范围。

---

### 2.2 环境 B：宿主机 / 弱隔离环境

**假设**：代码直接运行在用户宿主机上，一个恶意或误操作命令可能永久损坏系统、泄露密钥、删除生产数据。

**设计策略**：
- **内部安全检查**：作为第一道防线，**无条件硬拦截**
- **权限验证**：作为第二道防线，提供**可交互的策略控制**
  - 用户可配置 `Bash(rm:*)` deny 规则
  - 用户可配置 `Bash(git commit:*)` allow 规则
  - 不确定时 ask，用户可勾选"以后不再询问"

**执行流程**：

```
输入命令
  │
  ▼
[安全检查] ──► 硬约束，不可覆盖
  ├── 不通过 → 直接 deny
  └── 通过
        │
        ▼
[权限验证] ──► 用户策略，可覆盖
  ├── deny 规则命中 → deny
  ├── ask 规则命中 / 模式 ask → 弹窗 ask（用户可 allow）
  ├── allow 规则命中 → allow
  └── 无规则 → 读命令 ask / 读命令 allow（看模式）
        │
        ▼
[执行]
```

## 3. 为什么安全检查不能放在 `call` 里

一个常见的想法是：
> "安全检查是执行层面的东西，放在 `call` 方法里不就行了？"

**答案是：不行。**

### 3.1 交互语义问题

`call` 方法在统一框架中是在**用户已经批准（allow）之后**才被调用的。如果在 `call` 里发现 `eval` 或 `rm -rf /`，你只能：
- **直接抛异常**：用户体验极差（刚点了允许，突然就报错）
- **再 ask 一次**：需要重建一套弹窗/suggestion/classifier 机制，重复造轮子

而放在 `checkPermissions`（或一个等价的前置决策点）中，安全检查和权限规则可以**共享同一个 `PermissionResult` 交互协议**：
- 统一的弹窗
- 统一的 suggestion 生成
- 统一的投机分类器（speculative classifier）race
- 统一的审计日志

### 3.2 Hook 系统问题

`PreToolUse` Hook 在 `checkPermissions` 之前运行。如果安全检查下沉到 `call`，Hook 已经放行后才发现问题，Hook 失去了修正机会。

### 3.3 正确的做法

把 `checkPermissions` **内部**拆成两个阶段，但对外仍然只返回一个 `PermissionResult`：

```ts
async checkPermissions(input, context) {
  // 阶段 1：安全检查（硬约束）
  const securityResult = await runStaticSecurityChecks(input);
  if (securityResult.behavior === 'deny') {
    return securityResult; // 直接拒绝，不再查权限规则
  }

  // 阶段 2：权限规则（软约束）
  const permissionResult = await runPermissionChecks(input, context);

  // 合并优先级：security deny > permission deny > permission ask > allow
  return mergeResults(securityResult, permissionResult);
}
```

这样既保持了**内部职责清晰**，又享受了**统一交互框架**的所有能力。

## 4. 安全检查与权限验证的具体职责划分

### 4.1 内部安全检查清单（不可覆盖）

以下检查**任何情况下都不应被用户规则或模型覆盖**：

| 检查项 | 失败行为 | 原因 |
|--------|---------|------|
| AST `too-complex` | ask/deny | 无法静态证明安全 |
| `eval` / `source` / `exec` | deny | 动态执行入口，不可控 |
| zsh 危险 builtin（`zmodload` 等） | deny | 可扩展攻击面 |
| `rm -rf /` 类危险删除 | deny | 灾难性数据丢失 |
| `/proc/*/environ` 访问 | deny | 密钥泄露风险 |
| `jq --run-tests` / `jq system()` | deny | 代码执行桥接 |
| 输出重定向越界（如写 `/etc`） | deny/ask | 文件系统污染 |
| 管道中的 `cd + git` 组合 | ask | bare repo RCE |
| subshell `()` / command group `{}` | ask | 隐藏执行边界 |
| 命令注入特征（`$()`、`<()` 等） | ask/deny | 参数替换攻击 |

> 注：在宿主机环境中，`ask` 也是一种安全策略（fail-closed），但在强隔离环境中可直接 `deny`。

### 4.2 权限验证清单（用户可覆盖）

以下检查是**用户偏好**，可以被显式 allow/deny 规则覆盖：

| 检查项 | 行为 | 可覆盖性 |
|--------|------|---------|
| `Bash(git commit:*)` allow | allow | 用户可删除规则 |
| `Bash(rm:*)` deny | deny | 用户可删除规则 |
| 路径越界（如 `ls /home`） | ask | 用户可添加 `Read(/home/**)` allow |
| `cd` 到项目外目录 | ask | 用户可批准一次或加规则 |
| 复合命令含 `python` / `node` | ask | 用户可批准并保存规则 |

## 5. 模型增益（Classifier）的边界

AI 分类器（Prompt Classifier / Speculative Classifier）只能作用于**权限验证层**，**绝不能覆盖安全检查层**。

### 允许的模型增益

- 用户对某类命令的描述是"允许运行测试" → 分类器自动 allow `npm test`
- 用户之前批准过类似命令 → 投机分类器在弹窗出现前自动 allow

### 禁止的模型增益

- 即使分类器说"这看起来安全"，也不能放行 `eval` 或 `rm -rf /`
- 即使分类器说"这看起来安全"，也不能放行 AST `too-complex` 的命令

**原则**：
> 模型只能把 **ask/passthrough** 提升为 **allow**，不能把 **deny** 降级为 **ask/allow**。

## 6. 设计检查清单

在实现 Bash 工具（或类似高权限工具）时，请按以下清单自查：

- [ ] 是否有明确区分"安全检查"和"权限验证"的边界？
- [ ] 安全检查的结果是否有不可被覆盖的硬约束？
- [ ] 权限验证的结果是否允许用户通过弹窗或规则覆盖？
- [ ] `checkPermissions`（或等价决策点）是否在 `call` 之前执行？
- [ ] 安全检查与权限验证是否共享统一的 `PermissionResult` 交互协议？
- [ ] 模型分类器是否只能增益权限层，不能覆盖安全层？
- [ ] 在强隔离环境中，是否可以单独关闭权限验证但保留安全检查？
- [ ] 在弱隔离环境中，两层是否都完整生效？

---

## 7. 总结

1. **安全检查是底线**，防止已知灾难性后果，**不可被覆盖**。
2. **权限验证是偏好**，让用户控制工作流范围，**可被用户覆盖**。
3. **两层都应该在 `checkPermissions` 中完成**（对内分阶段，对外统一产出 `PermissionResult`），而不是把安全检查下沉到 `call`。
4. **环境决定策略**：Docker 可弱化权限层，宿主机必须两层齐开。
5. **模型只能增益权限层**，绝不能覆盖安全层。
